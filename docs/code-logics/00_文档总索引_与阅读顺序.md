# 微信绿植诊断小程序代码实现文档集：总索引

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、文档目标

这组文档不是产品 PRD，也不是粗略架构说明，而是把当前代码中已经存在的实现逻辑拆成可维护的工程文档。目标是让后续 Codex、人工开发、重构、审计和回归测试都能沿着同一套边界继续推进。

## 二、当前代码包边界

本次代码包主要包含 CloudBase 云函数后端：

- `diagnose-http`：诊断主服务，承担视觉接入、诊断运行时、问诊、排序、结果输出、审计和池外映射。
- `identify-http`：植物识别服务，调用百度植物识别能力并整理候选植物。
- `storage-http`：图片上传、临时访问 URL、诊断图片/植物图片存储路径管理。
- `weather-http`：天气上下文读取与缓存。
- `plant-user-http`：用户植物档案基础接口。
- `plant-catalog-http`：植物目录与映射接口。
- `auth-user-http`、`wechat-identity`、`wechat-phone`：用户身份、微信身份、手机号解密相关能力。
- `diagnosis-history-http`：历史诊断服务已退役，入口返回迁移说明。
- `layer`：共享工具层，包含 HTTP、CloudBase、LLM、配额、运行环境等公共封装。

未发现明确的小程序前端源码目录，例如 `src/pages`、`pages`、`components`、`app.config.ts`、`project.config.json` 之外的前端工程主体。因此本次不会编造前端内部状态管理、页面组件或 uni-app hook 的实际实现。

## 四、文档维护原则

- 修改诊断主链，必须同步修改第 3、4、5、6、7 份文档。
- 修改接口、返回字段、前端展示策略，必须同步修改第 2、7 份文档。
- 修改表结构、持久化、会话轮次，必须同步修改第 8 份文档。
- 修改 CloudBase、AI、微信小程序接入方式，必须同步修改第 10 份文档。
- 每次修改都应保留已有有效内容，只增不减；确有错误时用“废弃说明/修正说明”覆盖，而不是静默删除。
