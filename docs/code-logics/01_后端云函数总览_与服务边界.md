# 01 后端云函数总览与服务边界

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、整体服务拆分

当前后端是一组 CloudBase 云函数。诊断链路以 `diagnose-http` 为核心，其他云函数提供植物身份、图片、天气、用户植物和认证上下文。

| 服务 | 入口文件 | 当前职责 | 是否核心诊断链 |
|---|---|---|---|
| 诊断 HTTP 服务 | `cloudfunctions/diagnose-http/app.js` | 诊断开始、回答追问、结果读取、历史、反馈、审计、池外异常管理、兼容旧诊断接口、SSE 流式诊断 | 是 |
| 植物识别服务 | `cloudfunctions/identify-http/app.js` | 调用百度植物识别，返回植物候选、匹配状态和识别 ID | 是，诊断前置 |
| 图片存储服务 | `cloudfunctions/storage-http/app.js` | 上传诊断图片/植物图片、生成临时 URL、删除诊断图片 | 是，诊断前置 |
| 天气服务 | `cloudfunctions/weather-http/app.js` | 根据经纬度读取天气；优先按城市 `city` 做 24 小时共享缓存，并同步用户缓存供诊断读取 | 辅助诊断 |
| 用户植物服务 | `cloudfunctions/plant-user-http/app.js` | 用户植物档案基础接口 | 诊断上下文来源 |
| 植物目录服务 | `cloudfunctions/plant-catalog-http/app.js` | 植物目录、植物映射数据读取 | 诊断上下文来源 |
| 用户认证服务 | `cloudfunctions/auth-user-http/app.js` | 微信身份同步、用户资料读取/更新、邮箱/手机号绑定 | 基础能力 |
| 微信身份函数 | `cloudfunctions/wechat-identity/index.js` | 读取 CloudBase 注入的微信 openid/appid/unionid | 基础能力 |
| 微信手机号函数 | `cloudfunctions/wechat-phone/index.js` | 解密微信手机号 | 基础能力 |
| 历史诊断旧服务 | `cloudfunctions/diagnosis-history-http/app.js` | 返回迁移/废弃响应，历史能力已合并进 `diagnose-http` | 否 |

## 二、共享工具层

`cloudfunctions/layer` 是共享层，核心内容包括：

- `utils/http.js`：HTTP 请求解析、统一 JSON 响应、错误响应、用户信息解析、运行环境上下文。
- `utils/cloudbase.js`：CloudBase 初始化与复用。
- `utils/llm-request.js`：大模型请求封装。
- `utils/plant-diagnosis.js`、`utils/plant-knowledge.js`：植物诊断、植物知识相关通用方法。
- `utils/quota.js`：配额控制。
- `utils/runtime-env.js`：运行环境解析。

共享层的定位是“基础能力”，不应把具体诊断业务规则继续塞进 layer。诊断业务规则应留在 `diagnose-http/domain`、`diagnose-http/utils` 或 repository 内。

## 三、诊断服务内部模块分层

| 层级 | 目录 | 职责 |
|---|---|---|
| 入口层 | `app.js` | 解析 HTTP 路由、身份校验、调用服务/领域函数、组装前端最小响应、兼容旧接口、SSE 输出 |
| 领域层 | `domain/*` | 诊断主链、回答影响值、候选 outcome、route 路径规划、结果格式化、症状模式、非问题判断、不确定判断、高特异快速收敛 |
| 服务层 | `services/*` | 会话、运行时快照、追问队列、视觉诊断批次、最终化、读写状态、监督等编排 |
| 仓储层 | `repositories/*` | 从关系型数据库读取症状、问题、问题选项映射、先验、因果、审计、池外映射等数据 |
| 工具层 | `utils/*` | 视觉契约、诊断方向、派生证据、输出资格、上下文守卫、合成追问、公开摘要等纯逻辑 |
| 映射层 | `mappers/*` | 旧接口兼容、公开 ID 映射、数据回填/差异构造 |
| 展示层 | `presenters/*` | 将领域响应转成公开响应或紧凑响应 |
| 常量层 | `constants/*` | 路径选择参数、版本号、表名、字段形状、高特异规则、白名单等 |

## 四、关键边界原则

### 1. 入口层不承载核心诊断判断

`app.js` 可以做请求解析、身份、路由、持久化调用、响应裁剪，但不应继续扩写具体诊断规则。具体规则应进入领域层或工具层。

### 2. 领域层必须保持可审计

诊断规则必须能解释“为什么问这个问题”“为什么停止”“为什么输出不确定”“为什么不允许输出某个问题”。当前代码中已经通过 `decisionCause`、`followUpStopPolicy`、`outputEligibility`、`diagnosticTrace` 等结构承接审计信息。

### 3. 仓储层只负责数据读取和字段归一

仓储层可以处理 SQL、字段名、兼容字段，但不应承担业务判断。比如“真菌蚊不能由黄点自动带入”属于领域判断，不应藏在 SQL 查询里。

### 4. 前端只消费公开契约

后端内部字段很多，例如 `observedEvidenceSet`、`symptomClassRuntime`、`answerEffects`、`routeDecision`。前端不应直接依赖全部内部结构。正式页面应依赖 `buildFrontendDiagnosisResponse()` 和 presenter 输出的公开字段。
