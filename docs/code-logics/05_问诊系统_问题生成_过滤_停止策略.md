# 05 问诊系统：问题生成、过滤与停止策略

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、问诊系统定位

问诊是视觉证据无法安全闭合时的第二证据通道。它用于确认视觉候选是否真实存在、在多个病因之间分流、收集视觉看不到的养护上下文、对用户否定方向进行削弱、给输出资格补齐必需上下文。

## 二、问题来源

| 来源 | 说明 |
|---|---|
| 静态问诊表 | 数据库中已有的问题、选项、映射关系 |
| 症状模式题组 | 根据 `symptom_class` 选出的优先题组 |
| 动态合成追问 | 根据当前视觉候选或上下文临时生成的问题 |
| 视觉候选确认题 | 第一轮用于确认视觉模型看到的候选异常 |

相关代码包括 `repositories/question-repository.js`、`domain/question-selector.js`、`utils/synthetic-follow-up.js`、`utils/question-symptom-bridge.js`、`utils/question-target-dimension.js`。

## 三、问题选项映射

问题选项不是只保存用户选择文本，而是能映射成运行时效果：正向症状证据、负向症状证据、候选问题加分、候选问题惩罚、问题组完成状态、路径约束。因此每个选项都必须明确“确认什么、否定什么、削弱什么”。

## 四、首轮视觉候选追问

第一轮还没有正式观察症状但视觉候选存在时，系统会优先问确认题。比如疑似黄叶要问黄叶分布和叶位；疑似叶斑要问边缘、水渍感、扩散；疑似结构缺损要问虫体、虫粪、啃咬边缘、旧伤。这个机制避免视觉候选直接成为结论。

## 五、强制追问场景

即使排序分数看起来足够，以下场景也会强制追问：黄叶模式未分流、霉菌方向第一轮需要确认、上下文必需问题缺失、视觉候选与正式证据存在正交缺口、宽泛视觉异常未分流、水肿被用户否定为平斑后缺少独立叶斑证据、弱池外异常提示需要确认。

## 六、问题过滤顺序

候选问题不会直接展示。过滤包括：已问问题过滤、已确认视觉事实过滤、负向答案过滤、黄叶路径约束、已回答路径约束、产品口径裁剪为一个问题。

关键函数包括已回答路径约束过滤函数（代码函数：`filterFollowUpsByAnsweredRouteConstraints()`）、最终视觉确认题过滤函数（代码函数：`filterFinalVisualPresenceFollowUps()`）和追问停止策略评估函数（代码函数：`evaluateFollowUpStopPolicy()`）。

黄叶动态追问的已回答判断必须覆盖三类来源：当前回答、已落库追问记录、静态题目元数据。动态题如果没有静态题目行，系统要从题目代码名（代码字段：`questionKey`）反解目标维度（代码字段：`targetDimension`）和目标症状（代码字段：`targetSymptomKey`）。否则用户已经回答过“黄叶先出现在新叶还是老叶/下部叶”，系统仍可能误判为未回答并重复追问。

蜜露/黏腻分支有独立追问边界。蜜露/黏腻（代码症状：`sticky_honeydew`）不能继续复用泛化虫害类型维度（代码维度：`pest_trace_type`），因为用户可能已经在黄叶虫害分流里否定了细网或蓟马银白擦伤；再次展示同组泛化虫害类型题，会造成“刚否定又被拉回去”的体验问题。

蜜露/黏腻（代码症状：`sticky_honeydew`）也不能再动态追问表面黏腻维度（代码维度：`surface_stickiness`）。用户已经通过蜜露确认题回答“有”（代码问题与选项：`q_sticky_honeydew_confirm=yes`）后，系统不能再用另一个动态题重复确认同一黏腻事实。

黄叶虫害问题不能从已否定或非虫害分支回流。如果黄叶虫害痕迹类型（代码维度：`pest_trace_type`）已经回答“没有虫害痕迹”（代码选项：`no_pest_trace`），上下文守卫不得再强制蜜露确认题（代码问题：`q_sticky_honeydew_confirm`）。如果黄叶首要线索分流（代码维度：`yellowing_primary_clue_gate`）已经选择养护方向（代码选项：`care_context`）或病斑/霉层方向（代码选项：`disease_trace`），上下文守卫不得再把细网确认题（代码问题：`q_spider_webbing_visible`）、蓟马银白擦伤确认题（代码问题：`q_thrips_silver_streaks`）、蜜露确认题（代码问题：`q_sticky_honeydew_confirm`）拉回当前路径。

## 七、停止策略

停止策略关注：是否本来需要追问、是否还有可问问题、是否高特异快速收敛允许跳过、上下文守卫是否要求继续问、问题池是否耗尽。停止不代表一定输出具体问题，停止后仍可能输出合法不确定。

## 八、问题池耗尽

如果系统判断应该追问，但过滤后没有可用问题，就进入“问题池耗尽”状态。此时不能强行输出具体问题，必须看输出守卫是否允许。证据不足时应输出不确定。

## 九、问诊文案要求

面向用户的问题必须自然中文表达，不允许暴露内部术语。`symptom_class` 应表达为“当前看起来更像哪类异常”；`context guard` 应表达为“还缺少关键确认信息”；`output eligibility` 应表达为“当前证据还不足以直接判断”。

## 十、新增问题审计清单

新增任何问题必须补齐：问题 key、问题中文文案、选项中文文案、每个选项映射的症状或问题影响、正向/负向/未知语义、所属问题组、目标症状维度、是否视觉确认题、是否可被已确认视觉事实过滤、是否影响输出资格。

黄叶和蜜露相关新增问题还必须额外审计：是否会重复已回答的黄叶维度、是否会把“没有虫害痕迹”（代码选项：`no_pest_trace`）错误转成光照或营养加分、是否会在用户否定表面黏腻感（代码维度与选项：`surface_stickiness=no`）后仍闭合具体虫害、是否会在蜜露确认题回答“有”（代码问题与选项：`q_sticky_honeydew_confirm=yes`）后重复确认表面黏腻、是否会把养护或病斑/霉层分支重新拉回虫害确认题。
