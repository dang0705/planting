## 三、开发约束

1. 中文业务语义优先：所有诊断概念必须先有中文解释，再给代码字段。
2. 代码和文档同步：修改 `domain/diagnosis-engine.js`、`domain/outcome-route-planner.js`、`domain/outcome-route-planner-helpers.js`、`domain/result-formatter.js`、`domain/result-formatter-helpers.js`、`domain/symptom-classifier.js`、`domain/question-selector.js`、`utils/diagnosis-directions.js`、`utils/output-eligibility.js`、`utils/context-required-problem-guard.js`、`repositories/question-repository.js`、`services/session-*` 时必须同步更新文档。
3. 规则不能散落在入口文件：`app.js` 只做入口编排。
4. 数据库字段必须可审计：新增表字段必须说明中文名、英文字段名、类型、是否可为空、来源、参与环节、是否影响公开输出。
5. AI 输出必须经过结构化承接：模型输出必须经过结构化解析、症状池映射、正式证据准入、诊断方向归纳、输出守卫。

## 四、审计清单

### 修改视觉标签时

检查是否有中文定义、是否映射到症状池、是否能进入正式证据、是否仅作为候选、是否影响高特异快速收敛、是否需要池外机制。

### 修改问题时

检查是否有问题组、目标症状维度、选项语义、unknown 选项、是否会被已确认视觉事实过滤、是否会产生负向证据。

黄叶和蜜露相关问题还要检查：蜜露确认题回答“有”（代码问题与选项：`q_sticky_honeydew_confirm=yes`）后，不能再重复追问表面黏腻维度（代码维度：`surface_stickiness`）；黄叶虫害痕迹类型（代码维度：`pest_trace_type`）回答“没有虫害痕迹”（代码选项：`no_pest_trace`）后，不能再强制蜜露确认题（代码问题：`q_sticky_honeydew_confirm`）；黄叶首要线索分流（代码维度：`yellowing_primary_clue_gate`）选择养护方向（代码选项：`care_context`）或病斑/霉层方向（代码选项：`disease_trace`）后，不能再强制细网、蓟马银白擦伤或蜜露这类虫害问题。

### 修改排序时

检查是否改变最终分数、top1、输出资格、低置信或不确定、是否有批跑覆盖。

### 修改输出时

检查不确定结果是否隐藏内部 topProblem、非问题结果是否被池外异常阻断、具体问题是否在诊断方向范围内、行动建议是否与结果类型匹配。`uncertain` 必须不回填 topProblem，`non_problematic` 不允许给出治疗类 actionAdvice。

黄叶和蜜露相关输出还要检查：整叶黄化（代码症状：`uniform_yellowing`）不能单独闭合缺氮或营养不足；“没有虫害痕迹”（代码选项：`no_pest_trace`）不能给光照不足、晒伤或营养不足加分；已有表面黏腻感不存在（代码维度与选项：`surface_stickiness=no`）证据时，不能闭合红蜘蛛、蚜虫、白粉虱或蓟马等具体虫害。
`route` 已接管 `followup` 与 `visibleOutcomes/actionAdvice` 时，默认不得再把 `ranking` 作为最终公开结论主链，仅保留作为候选排序与 fallback 回退输入。黄叶问题收敛时，后端以页内 `answers` 分组累计为准，`yellowing_care_area_gate` 仅表示问题分组标识，不再作为单点聚合 gate 做闭合判定。

### 修改会话时

检查旧会话是否还能读取、快照是否能解码、旧问题能否被错误回答、answer revision 是否正确推进。

## 五、上线前必须跑的检查

1. `route` 接管后，`ranking` 降级为候选排序和 fallback 审计；`routeDecision` 只保留最小公开字段，不对外暴露 `routeTrace/gateResults/internal ranking`。
2. `diagnose-route-regression-runner` 作为正式 route 回放验收口径，并至少保持三类真实样本 `3/3` 通过；本地 shell 回放只作为辅助验证。

第一轮无正式证据但有视觉候选必须问确认题或重拍/不确定；黄叶必须分流；孔洞必须分流；宽泛斑块必须追问或不确定；池外无映射必须保守不确定；用户否定某方向不能跳到缺证方向；不确定公开层不得出现具体最高问题；高特异快速收敛必须有正式证据和输出资格；历史读取能看到最终公开响应；反馈提交能绑定 session/result。

最新全量零模型穷举（代码策略：`engine-exhaustive-count`）基线：28 个模式、360 条终止路径、0 个失败、最大问诊深度 5。黄叶和蜜露硬性审计项必须保持为 0：重复问题（代码审计项：`repeatedQuestion`）、无虫害痕迹却输出虫害（代码审计项：`noPestButPestOutcome`）、蜜露确认后重复问表面黏腻（代码审计项：`stickyConfirmThenRepeatSurface`）、黏腻否定后闭合具体虫害（代码审计项：`stickyDeniedConcrete`）、养护分支被拉回虫害题（代码审计项：`careBranchPulledToPestQuestion`）、病斑/霉层分支被拉回虫害题（代码审计项：`diseaseBranchPulledToPestQuestion`）。

黄叶前置养护题还必须保留 0 模型 smoke 覆盖：`yellowing_care_area_gate` 下的一页一组问题内 12 个选项都要能显示题干、显示 SQL 来源选项文案、保留 `1-2` 这类横杠文本；浇水/光照/施肥/通风湿度等前置上下文选项应直接闭合到对应 outcome（由累计 `answers` 驱动），稳定/未知选项才继续叶龄和分布分流。特别检查 `airflow_humidity_rapid_spreading` 不得直接输出 `uncertain`。
