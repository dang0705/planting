# 03 诊断运行时主链路：逐步执行逻辑

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、主入口

诊断运行时主入口是 `cloudfunctions/diagnose-http/domain/diagnosis-engine.js` 的 `runDiagnosisRound()`。它不是单纯“根据症状查问题”，而是一轮完整推理编排，负责植物上下文、视觉证据、用户问诊答案、症状模式、诊断方向、候选 outcome、回答影响值、route 路径规划、追问生成、停止策略、输出资格、不确定结果、非问题结果、高特异快速收敛和前端公开响应。

## 二、输入参数

| 入参 | 中文含义 | 用途 |
|---|---|---|
| `openid` | 当前微信用户身份 | 读取用户植物、天气、会话等用户级数据 |
| `plantId` | 植物 ID | 查询植物上下文 |
| `userPlantId` | 用户植物 ID | 查询用户养护档案 |
| `lockedPlantContext` | 锁定植物上下文 | 多轮诊断中避免植物身份漂移 |
| `observedSymptoms` | 旧版观察症状 | 兼容旧链路，会被转成正式观察证据 |
| `observedEvidenceSet` | 正式观察证据集合 | 运行时推理的核心证据集合 |
| `visualAggregateResult` | 视觉聚合结果 | 图片模型输出后的统一视觉结果 |
| `answers` | 当前已有问诊答案 | 转成问诊证据、负向证据、路径约束 |
| `askedQuestionKeys` | 已问问题 key | 避免重复问、判断问诊历史 |
| `answeredQuestionGroupKeys` | 已回答问题组 | 控制症状模式和问题组持续优先级 |
| `unknownCountByGroup` | 每组“不确定”次数 | 防止同一方向反复问无效问题 |
| `symptomClassState` | 上一轮症状模式状态 | 多轮继承症状模式 |
| `round` | 当前轮次 | 控制是否还能追问、是否第一轮视觉候选追问 |
| `stage` | 当前阶段 | 区分 preliminary、followup、final |
| `sessionId` | 诊断会话 ID | 读取/保存追问队列和运行时快照 |

## 三、执行总流程

### 第 1 步：解析或读取植物上下文

如果传入 `lockedPlantContext`，运行时优先使用锁定上下文，防止多轮诊断中植物身份变化。否则根据 `openid`、`plantId`、`userPlantId` 读取植物信息。植物上下文包括植物显示名、身份 ID、身份解析状态、最近视觉批次 ID、属、科、类目、浇水、施肥、光照、通风、温湿度区间、养护基线审核状态和变异程度。

### 第 2 步：解析视觉路由上下文

`resolveVisualRouteContext()` 从视觉聚合结果中提取视觉建议的主动作、视觉路由提示、建议追问捕获点和原始视觉调用批次 ID。视觉路由只提供流程方向，不直接决定最终诊断结论。

### 第 3 步：读取已回答问题的选项映射

系统收集当前答案里的题目代码名（代码字段：`questionKey`），读取数据库问题选项映射，并合并合成追问选项映射（代码函数：`buildSyntheticFollowUpOptionMappings()`）。这些映射用于把用户回答转成正向症状证据、负向症状证据、问题加权调整和路径约束。

黄叶动态追问有额外要求：当前回答（代码字段：`answers`）必须并入已回答题目集合。动态生成的题目如果没有从静态问诊表读回完整题目行，运行时要从题目代码名（代码字段：`questionKey`）反解目标维度（代码字段：`targetDimension`）和目标症状（代码字段：`targetSymptomKey`）。这样系统才能知道“黄叶新老叶分流”“黄叶分布分流”等维度已经回答过，不能下一轮重复追问。

### 第 4 步：合并正式观察证据

运行时合并四类证据：已有 `observedEvidenceSet`、旧版 `observedSymptoms` 转换出的证据、视觉聚合结果转换出的证据、用户回答映射出的追问种子证据。合并函数是 `mergeObservedEvidenceSet()`。证据进入推理必须满足业务口径：进入正式观察证据集合，并且 `enteredRuntime=1`。

### 第 5 步：补齐症状字典标签

系统收集证据中的 `symptomKey`，读取症状字典，再将字典字段回填到证据和观察症状中。这样后续逻辑能拿到中文名、症状类别、形态维度、问题映射等信息。

### 第 6 步：生成派生证据

`buildDerivedEvidenceSet()` 根据正式观察证据和症状字典生成派生证据。派生证据不是用户或视觉直接说出来的事实，而是由已确认事实推导出来的运行时辅助信号。它可以增强方向判断，但不能替代正式观察证据直接越级输出。

### 第 7 步：收集视觉候选症状

`collectVisualCandidateSymptoms()` 从视觉聚合结果中提取视觉候选。视觉候选不等于正式证据，主要用于第一轮 seed follow-up，即“图片看起来可能是什么，需要问用户确认”。

### 第 8 步：归纳诊断方向

`buildDiagnosisDirections()` 结合正式证据、派生证据、视觉候选和视觉路由提示，生成诊断方向。诊断方向用于控制候选问题和输出范围，避免最高排序问题跑到无关方向。

### 第 9 步：解析视觉症状模式

`resolveSymptomClassRuntime()` 根据当前观察症状、轮次、已回答问题组、不确定次数和上一轮状态，得到症状模式运行时状态。症状模式用于决定当前更像黄叶、叶斑、结构损伤、生长异常、虫害等哪一类，并控制问题组和输出守卫。

### 第 10 步：处理池外视觉异常

如果视觉聚合结果属于“池外异常且没有已审计 proxy mapping”，系统直接输出保守不确定，而不是强行归入现有问题池。看不懂的异常不能硬塞进已有问题。

### 第 11 步：非问题性判断

`resolveNonProblematicRule()` 判断当前是否满足非问题规则。但如果存在弱池外异常提示，则会阻断非问题结论，避免“看到了异常但系统没收录”时错误安抚用户。

### 第 12 步：第一轮视觉候选追问

如果第一轮还没有正式观察症状，且视觉不要求重拍，系统会优先生成视觉候选确认题。黄叶模式会优先走黄叶分流题，而不是让视觉候选直接闭合。

### 第 13 步：构建候选问题先验

`buildCandidatePriors()` 根据植物上下文和观察症状构建候选问题先验。随后合并诊断方向覆盖补充、用户答案直接调整、属级兼容性和宿主兼容性。候选池为空时必须输出不确定。

### 第 14 步：计算证据分和惩罚项

`computeVisualEvidenceScores()` 计算视觉/观察证据支持度；`computeQuestionEvidenceAndPenalty()` 计算用户问诊答案支持、削弱和惩罚；`computeGenusFactor()` 与 `computeHostFactor()` 调整属级和宿主兼容。最终排序是多个证据来源的综合结果。

### 第 15 步：解析高特异快速收敛计划

`resolveHighSpecificityConvergencePlan()` 判断是否存在高特异视觉证据，是否允许减少或跳过追问。高特异视觉证据只能加快收敛，不能绕过正式证据集合、输出资格和守卫。

### 第 16 步：判断是否必须追问

系统综合排序稳定性、top1 输出资格、分数差距、视觉路由、上下文必需问题、霉菌方向、视觉候选缺口、黄叶分流、宽泛视觉异常、弱池外异常等原因决定是否追问。

在 `route` 接管模式下，追问只允许来源于 `route` 主链：
- 每轮最多只保留 1 个追问；
- 全局最多 2 轮，超过预算时不再强补追问，转入保守兜底；
- 不再执行旧 `generic/forced follow-up` 的兜底补位逻辑。
- 黄叶养护 route 已去除 `yellowing_care_area_gate` 聚合闭合口径，改为“一页一组分组问题”展示：每页先收集同组可理解问题（浇水、光照、施肥、通风/湿度），后端再按组累计 `answers`，由 `route` 的 `gate` 与 `outcome` 收敛决策。
- 黄叶养护 route 的单个上下文题不能直接闭合 outcome。即使某个分组选项能支持具体 route，例如 `watering_frequency_context=often_wet/often_dry`，也必须先完成黄叶必答分组（浇水、光照、施肥、通风/湿度发展速度）后，才能公开 `visibleOutcomes/primaryOutcome`。`/diagnosis/answer` 的 route fast path 也必须复用同一 hold 规则，不能早于主链守卫直接返回 `final`。
- `routeDecision.nextQuestionKeys` 生成候选题时必须过滤组内已闭合事实。黄叶问题在同组内由累计 `answers` 决定，不以“是否已回答某单题 `questionKey=yellowing_care_area_gate`”作为唯一出口；只有当组内已达闭合条件时才可推进到未满足的叶龄、分布等分流题，不能重复回到已闭合分组，也不能在前置上下文已闭合时额外追问。
- 诊断 review 后台详情必须保留审计用完整 route path：`runtime_snapshot_json.routeDecision` 应映射到 `coreProcess.route.routeDecision`，包括 candidateOutcomeStates、routeTrace、gateResults、nextQuestionKeys、decisionCause 和 action conflict groups；用户端公开结果仍只展示净化后的 route 摘要。

### 第 17 步：生成、过滤、裁剪追问

候选追问先生成，再经过黄叶路径过滤、已确认视觉事实过滤、已回答路径约束过滤、问题去重、产品口径裁剪为一个问题、停止策略判断。黄叶路径过滤必须使用“静态字段优先、题目代码名兜底反解”的同一口径识别已回答维度，不能只依赖静态题目行。

### 第 18 步：不追问时进入输出守卫

不再追问时不会直接拿最高分输出，而是继续执行输出资格检查、诊断方向范围检查、上下文必需守卫、黄叶未分流守卫、结构损伤未分流守卫、叶斑桥接路由缺口守卫、宽泛视觉异常未解决守卫、症状模式与问题族不一致守卫。

### 第 18a 步：route 决策接管主链

当前收口后，`route` 已成为主链路收敛机制：

1. 运行时不再生成或消费 `ranking` 作为候选排序、fallback 顺序或审计依据；候选集合只保留 `candidateOutcomes` 与 route gate 状态。
2. `routeDecision.nextQuestionKeys` 作为追问主接管路径，`followup` 优先按其主链执行。
3. `primaryOutcome`、`secondaryOutcomes`、`visibleOutcomes`、`actionAdvice` 按 route-aware 输出链生成，只有满足公开资格后才可下发前端。
4. `routeDecision` 公开仅保留最小摘要；`routeTrace`、`gateResults` 等内部明细只进入 review/debug，不作为用户端稳定字段。
5. 最终公开展示优先从 `visibleOutcomes`（多 outcome）落到前端，不可再只取 `finalResult` 的最后一次主问题；`finalResult` 仅作为主结论与兼容下发字段保留。

6. 当命中组合症状输入时，先做 symptom canonicalize（如 `wilting_wet_soil`、`wilting_dry_soil` 先映射为 `wilting` 与 `soil_wet`、`soil_dry` 原子症状）后，再进行 `route` 分组与 gate 匹配，避免 0 模型 smoke 中出现 `route_no_visible_outcomes_for_route/uncertain` 的回退。
7. 路由可见 outcome 上限取值改为：先取所有 `activeRouteGroups` 的可见上限中可用正数最大值，再受全局 `maxVisibleOutcomes` 上限截断；避免低上限组把黄叶等高意图组压回 1/2。
8. `action conflict` 只用于 `actionAdvice` 层面的冲突保护，不应清空 `secondaryOutcomeKeys`；`visibleOutcomeKeys` 与 `secondaryOutcomeKeys` 在冲突场景下保留用于 route 公开输出与前端展示链路。
9. 所有 route fast path 只能减少 DB 和主链计算开销，不能绕过用户可见输出守卫。黄叶模式下，fast path 命中 route gate 前必须先检查必答分组是否完成；未完成时只能返回 `null` 交回主链继续追问。

#### 18a-1 路由 answer effects 覆盖与 fail-soft

`/diagnosis/answer` 的 route fast path 只允许把 `preloadedRouteAnswerEffects` 当作缓存提示，不得将其视为完整 `route` 历史集合。  
fast path 触发前必须先做完整历史覆盖检查：

1. 以完整历史 `routeAnswerEffectQuestionKeys` 作为覆盖集；
2. 逐项比对 `preloadedRouteAnswerEffects` 是否命中；
3. 对缺失 questionKeys，按 `outcomeRouteRepository.getOutcomeAnswerEffects(missingQuestionKeys)` 补查并做去重合并；
4. 合并结果作为后续 route fast path 决策输入，保证不会因 preload 不足截断历史候选。

所有 `routeAnswerEffects` 查询与合并失败均采用 fail-soft：失败时不抛异常、不终止请求、返回 `routeAnswerEffects=[]`，并继续走完整路径；完整路径也复用同一 fail-soft 逻辑，避免 `/diagnosis/answer` 发生 500 回退。

该规则的验收口径：route fast path 在缺失历史效果记录时，仍能正确返回 `follow-up`/`final`（按主链守卫），且不得因为 effects 覆盖不足导致 premature final 或报错。
### 第 18b 步：route-aware 公开守卫与兜底

在 route 接管输出链后，最终公开必须再经过两个底线守卫：

1. `uncertain` 下不在公开层回填 `topProblem`/`topProblemKey`，不回填 `finalResult.primaryOutcome`/`primaryOutcome`，只保留合法的不确定说明与保守建议。
2. `route` 非权威（`fallbackPolicy` 非空）时，公开路径统一兜底为 `uncertain`，且不回填具体 `finalResult/problem` 结果。
3. `non_problematic` 下不产出治疗型或病害型 `actionAdvice`。
4. 当 `route` 不权威时，`fallbackPolicy` 当前只允许收敛为 `uncertain`；不得再回退到旧排名候选来生成具体结论。
5. `visibleOutcomeKeys` 存在且 `visibleActionConflictGroups.length > 1` 的冲突路由场景下，不清空 `primaryOutcomeKey`，也不下沉为 `outcomeType=uncertain/route_uncertain_with_candidates`；应保留排序后的主可见 outcome，用 route-backed finalResult 作为公开输出，并在 `actionAdvice` 中设置 `conflictDetected=true`，`nextSteps` 仅携带 `route_conflict_guard`，不回退到 `careGuidance` 的单边浇水/施肥/用药建议。
6. 当 `route` 权威且非 `uncertain` 输出时，`explanation/resultExplanation` 必须与 `routeDecision.primaryOutcomeKey` 对齐，不能沿用旧候选问题主项的说明来源；若 route 主体无说明项，用 `primaryOutcome summary/firstAid/avoid/routeSafeSummary` 做保守兜底。

### 第 19 步：生成停止决定

`stopDecision` 描述最终为什么停止：`outcomeLocked`、`stopReason`、`uncertainLegalityReason`、`stopReasonDetail` 和可审计的 `decisionCause`。

### 第 20 步：格式化公开结果

`formatDiagnosisResponse()` 最终输出公开结果。结果类型主要包括等待追问、问题性、合法不确定、非问题性。

## 四、运行时主链维护红线

1. 视觉原始候选不能直接越级输出问题。
2. 用户否定某方向，只能削弱该方向，不能自动跳到另一个缺证方向。
3. 宽泛视觉异常必须追问或输出合法不确定。
4. 黄叶和结构损伤必须先做病因分流。
5. 不确定结果不能在公开层回填具体最高问题；`non_problematic` 不允许输出治疗型建议。
6. 高特异快速收敛不能绕过输出资格和正式证据集合。
7. 黄叶动态追问必须把当前回答纳入已回答题目判断，动态题缺少静态元数据时必须能从题目代码名反解目标维度和目标症状。
8. 性能优化路径不得绕过诊断主链守卫；新增或改造 fast path 时必须补“只答首个黄叶浇水分组题不得 final”的回归用例，并跑真实 `/diagnosis/answer` smoke。

## 本轮 route-only 收敛补充（2026-05-17）

- 活跃主链路仅保留 `route 路径规划` 的收敛机制：候选 outcome 的生成、过滤与 `路径 gate` 评估后，走 `routeDecision` 决定 `final / follow-up` 走向。
- 禁止再以 `ranking/rankings/finalScore/baseScore/scoreDelta/topScoreGap` 作为主链路决策输入或 fallback 依据；历史字段不得回填到 runtime/public payload。
- `answerEffects.effectValue` 是“回答影响值”，只解释答案与候选 outcome 的证据关系，不参与排序或分差判断。
- `review/detail` 与 `review/list` 不再要求看到分数排名；review 只展示 route path、gate、候选 outcome 与证据说明。
