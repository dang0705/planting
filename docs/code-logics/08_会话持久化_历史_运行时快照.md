# 08 会话持久化、历史与运行时快照

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、为什么需要会话持久化

诊断不是一次性请求。用户可能上传图片、获得第一轮追问、回答问题、进入第二轮追问或最终结果、查看历史、提交反馈。因此系统必须保存会话状态、问题队列、运行时证据、轮次结果和最终结果。

## 二、主要服务文件

| 文件 | 职责 |
|---|---|
| `services/session-service.js` | 会话主服务，构建 session、读取状态、保存历史、反馈、答案标记等 |
| `services/session-service-helpers.js` | 会话辅助函数 |
| `services/session-read-service.js` | 会话读取 |
| `services/session-runtime-write-service.js` | 运行时写入 |
| `services/session-state-write-service.js` | 会话状态写入 |
| `services/session-finalization-service.js` | 会话最终化 |
| `services/session-follow-up-service.js` | 追问相关会话操作 |
| `services/session-runtime-snapshot-codec.js` | 运行时快照编码/解码 |
| `services/round-runtime-persistence-service.js` | 每轮运行时持久化 |
| `services/question-queue-runtime-service.js` | 问题队列状态管理 |
| `services/session-supervision-service.js` | 会话监督和异常状态处理 |

## 三、会话核心状态

会话至少应包含用户 openid、`sessionId`、当前轮次、当前阶段、锁定植物上下文、最近视觉批次 ID、已观察证据集合、已问问题列表、已回答问题列表、已回答问题组、每组未知次数、症状模式状态、当前结果 ID、是否最终化。

## 四、轮次状态

每轮应保存 `roundId`、`round`、`stage`、输入证据、合并后证据、派生证据、诊断方向、问题排序摘要、追问问题、停止决定、输出结果、视觉批次关联。这些信息用于复盘为什么当时问了这个问题或输出了这个结果。

## 五、问题队列

`question-queue-runtime-service.js` 负责标记队列项已回答、使旧轮次问题失效、保证用户回答的是当前有效问题。这层很关键，因为前端可能重复提交，用户可能返回旧页面，网络可能重试。

## 六、答案版本

`prepareAnswerRevision()` 和 `resolveNextAnswerRevision()` 用于推进答案版本。答案版本用于区分同一会话中的多次提交、避免重复提交覆盖状态、支持审计“第几版答案导致了本轮变化”。

## 七、运行时快照

运行时快照应保存正式观察证据、派生证据、症状模式状态、诊断方向、追问状态、输出资格、停止原因。快照解码失败时，不能继续用半残状态输出具体问题，应降级为可解释错误或保守不确定。

## 八、历史与反馈

`diagnose-http` 已接管 `/diagnosis/history`、`/diagnosis/result`、`/diagnosis/feedback`。`diagnosis-history-http` 已退役，后续不要在旧服务上新增功能。

## 九、持久化维护要求

新增运行时关键字段时同步快照编码/解码；新增追问状态时同步问题队列和答案校验；新增输出结果字段时同步历史详情读取；新增审计字段时同步 review detail；删除字段必须保留兼容读取逻辑。
