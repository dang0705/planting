# 07 结果格式化、公开响应与前端接入契约

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、结果格式化入口

核心文件是 `cloudfunctions/diagnose-http/domain/result-formatter.js`，核心函数是 `formatDiagnosisResponse()`。它把诊断领域层的复杂对象转成统一公开响应。

## 二、结果类型

| 类型 | 中文含义 | 前端行为 |
|---|---|---|
| `await_follow_up` | 等待追问 | 展示一个追问问题和选项 |
| `problematic` | 有具体问题结论 | 展示问题名称、原因解释、建议步骤、避免事项 |
| `uncertain` | 合法不确定 | 展示无法确认原因、建议补图/观察/保守处理，不展示具体 topProblem |
| `non_problematic` | 非问题性状态 | 展示正常或非病害说明，给轻量养护建议 |

## 三、前端最小响应

### 3.1 三层响应模型

1. `engine/result` 内部层
   来源：`diagnosis-engine` 与 `result-formatter`。本层是内部运行对象与计算结果，包含 `routeDecision` 原始细节、`ranking`、`routeTrace`、`gateResults` 等，不是前端稳定依赖。

2. `route debug` 摘要层
   来源：对内部层最小化裁剪后的审计摘要。字段以 `routeDecision` 为主，当前公开净化后的实际口径是：`mode`、`visibleOutcomeKeys`、`primaryOutcomeKey`、`secondaryOutcomeKeys`、`activeRouteGroupKeys`、`nextQuestionKeys`、`fallbackPolicy`、`decisionCause`。该层用于开发/复查，不作为前端主契约。
   其中 `fallbackPolicy` 非空表示 `route` 非权威，且当前语义收口为 `uncertain`，用于 `debug/review` 区分回退路径；该层不与前端稳定契约混用。

3. `presenter` 公开层
   最终给前端消费的是 `diagnosis-round-presenter.js` 的最终公开对象（或其兼容版本）。前端稳定依赖应以 `diagnosis-round-presenter.js` 输出为准，不在本层复用内部 debug 字段。

`buildFrontendDiagnosisResponse()` 主要作用是做内部层到 `route debug` / `presenter` 的输入整理；前端字段口径以 `diagnosis-round-presenter.js` 为最终源，不再把 `routeDecision`、`visual_discriminators`、`missing_info_for_path`、`sessionId`、`currentRoundIndex`、`adviceSteps`、`avoidAdvice` 作为 presenter 的稳定顶层契约字段。

## 四、追问页面契约

当 `followUpRequired=true` 或 `outcomeType=await_follow_up` 时，前端应展示当前问题和选项。用户选择后调用 `/diagnosis/answer`，携带 `sessionId`、`roundId`、`questionId`/`questionKey`、`optionId`/`optionKey`。禁止前端自行推断结果，也不要把后端单题流程改成多题合并提交。

## 五、最终结果页面契约

问题性结果展示问题名称、证据说明、原因解释、建议步骤、避免事项和观察周期。不确定结果展示无法稳定判断的原因、补图/观察/保守处理建议，禁止展示内部 `topProblem` 或 `topProblemKey`。非问题性结果展示为什么暂不认为是病虫害或养护问题，以及需要继续观察的变化。

## 六、行动建议绑定方式

当前行动建议主要由结果格式化、问题解释、低置信建议和非问题规则共同生成。建议后续收敛为：具体问题级建议、问题簇级建议、不确定原因级建议、非问题状态建议、重拍建议。不要把所有建议都硬绑定到具体问题，否则“不确定但方向明确”的场景会缺建议。

## 七、公开字段与内部字段边界

前端可展示中文标题、摘要、说明、建议、选项、公开问题 ID、公开选项 ID、视觉摘要、重拍建议、会话与轮次 ID。前端不应展示 `rankings`、`answerEffects`、`symptomClassRuntime` 私有结构、内部 `decisionCauseDetails` 全量对象、数据库表字段原名、模型原始输出。

### 7.1 route-aware 公共响应边界

1. `routeDecision` 是 debug/审计最小摘要，默认不作为前端通用稳定字段。当前净化后的实际字段为：`mode`、`visibleOutcomeKeys`、`primaryOutcomeKey`、`secondaryOutcomeKeys`、`activeRouteGroupKeys`、`nextQuestionKeys`、`fallbackPolicy`、`decisionCause`。不向用户端暴露 `routeTrace`、`gateResults`、`rankings`、`rawRankingDebug`。
2. `primaryOutcome`、`visibleOutcomes`、`secondaryOutcomes`、`actionAdvice` 按 `route` 主链路输出，`ranking` 只作为排序与补位依据，不做最终公开结论主链。
3. `uncertain` 下不回填 `topProblem`/`topProblemKey`、`finalResult.primaryOutcome`/`primaryOutcome`，避免把 `route`/`ranking` 内部状态泄露到结果页。
4. `non_problematic` 不产生治疗型 `actionAdvice`，`uncertain` 不展示 `topProblem/topProblemKey`，仅给保守观察与补图建议。
5. `route` 模式下不向前端输出 `rankings`，由 `rankingAudit`（或 debug/review 快照）承载排序审计需求；前端只消费 `debug` 摘要与 presenter 公开字段。
6. `visual_discriminators` 与 `missing_info_for_path` 若存在，应进入 route debug 摘要与 presenter 归一化过程，作为可解释说明输入，而非直接替代最终业务结论。

## 八、验收口径与回放约束

- `diagnose-route-regression-runner` 已形成正式验收口径：三类真实样本云内回放 `3/3` 通过。
- 本地 `shell` 回放脚本保留为开发辅助，不作为 route 主链路完成的唯一判定条件；环境 DNS/fetch 波动导致的失败不直接等价于业务回归失败。

## 九、旧接口兼容

`toLegacyCompatiblePayload()` 和 `legacy-diagnose-presenter.js` 用于兼容旧诊断调用。新前端不建议继续依赖旧格式。

## 十、前端缺失源码情况下的接入建议

后续补前端源码后，应增加诊断首页上传图片流程、图片压缩与 `uploadCompression` 字段写入、诊断开始请求 payload、追问页状态机、结果页分支展示、历史与反馈页、错误态、重拍态、弱网态、SSE 进度展示等文档。
