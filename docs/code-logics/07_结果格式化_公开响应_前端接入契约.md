# 07 结果格式化、公开响应与前端接入契约

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、结果格式化入口

核心文件是 `cloudfunctions/diagnose-http/domain/result-formatter.js`，核心函数是 `formatDiagnosisResponse()`。它把诊断领域层的复杂对象转成统一公开响应，展示类纯逻辑已抽到 `cloudfunctions/diagnose-http/domain/result-formatter-helpers.js` 中。

## 二、结果类型

| 类型 | 中文含义 | 前端行为 |
|---|---|---|
| `await_follow_up` | 等待追问 | 展示一个追问问题和选项 |
| `problematic` | 有具体问题结论 | 展示问题名称、原因解释、建议步骤、避免事项 |
| `uncertain` | 合法不确定 | 展示无法确认原因、建议补图/观察/保守处理，不展示具体 topProblem |
| `non_problematic` | 非问题性状态 | 展示正常或非病害说明，给轻量养护建议 |

## 三、前端最小响应

### 3.1 三层响应模型

1. `engine/result` 内部层
   来源：`diagnosis-engine` 与 `result-formatter`。本层是内部运行对象与计算结果，包含 `routeDecision` 原始细节、`routeTrace`、`gateResults` 等，不是前端稳定依赖。

2. `route debug` 摘要层
   来源：对内部层最小化裁剪后的审计摘要。字段以 `routeDecision` 为主，当前公开净化后的实际口径是：`mode`、`visibleOutcomeKeys`、`activeRouteGroupKeys`、`nextQuestionKeys`、`fallbackPolicy`、`decisionCause`。该层用于开发/复查，不作为前端主契约；不再输出 `primaryOutcomeKey/secondaryOutcomeKeys`。
   其中 `fallbackPolicy` 非空表示 `route` 非权威，且当前语义收口为 `uncertain`，用于 `debug/review` 区分回退路径；该层不与前端稳定契约混用。

3. `presenter` 公开层
   最终给前端消费的是 `diagnosis-round-presenter.js` 的最终公开对象（或其兼容版本）。前端稳定依赖应以 `diagnosis-round-presenter.js` 输出为准，不在本层复用内部 debug 字段。

`buildFrontendDiagnosisResponse()` 主要作用是做内部层到 `route debug` / `presenter` 的输入整理；前端字段口径以 `diagnosis-round-presenter.js` 为最终源，不再把 `routeDecision`、`visual_discriminators`、`missing_info_for_path`、`sessionId`、`currentRoundIndex`、`adviceSteps`、`avoidAdvice` 作为 presenter 的稳定顶层契约字段。

## 四、追问页面契约

当 `followUpRequired=true` 或 `outcomeType=await_follow_up` 时，前端应展示当前问题和选项。用户选择后调用 `/diagnosis/answer`，携带 `sessionId`、`roundId`、`questionId`/`questionKey`、`optionId`/`optionKey`。禁止前端自行推断结果，也不要把后端单题流程改成多题合并提交。

## 五、最终结果页面契约

问题性结果展示问题名称、证据说明、原因解释、建议步骤、避免事项和观察周期。不确定结果展示无法稳定判断的原因、补图/观察/保守处理建议，禁止展示内部 `topProblem` 或 `topProblemKey`。非问题性结果展示为什么暂不认为是病虫害或养护问题，以及需要继续观察的变化。

## 六、行动建议绑定方式

`route` 权威且输出非 `uncertain` 结论时，`actionAdvice` 是用户可见行动建议的主来源。`nextSteps`、`whatToAvoid`、`treatmentText`、`preventionText` 只是面向旧前端或历史字段的投影与兼容 fallback，不能反向覆盖 `route` outcome 绑定的 `actionAdvice`。

自本轮修复起，`resolveRouteOutcomePayload` 输出需显式按 `visibleOutcomes[*]` 拆分并透传建议子项：
- `visibleOutcomes[*].actionAdviceItems`
- `visibleOutcomes[*].avoidAdviceItems`

`compact/minimal frontend response` 与 `normalizeDiagnosisResult` 均应保留并透传上述两个数组；前端不应将 `actionAdvice` 作为唯一建议来源，而应按每个 outcome 分组消费 `actionAdviceItems` / `avoidAdviceItems`，再按分组展示“治疗建议/避免事项”。

旧逻辑中的问题解释、低置信建议、`explanation.firstAid`、`explanation.avoid` 仅在缺少结构化 `actionAdvice` 或合法 `uncertain` 场景下兜底。低置信补救建议不得注入已经由 `route` 权威闭合的具体问题结论，否则会出现最终 outcome 与处理建议来自不同病因的问题。

`result-formatter` 约束：当 `route` 是权威且非 `uncertain` 时，`explanation`/`resultExplanation` 需跟随 `visibleOutcomes[0]` 或等价可见 outcome 的问题与说明来源，不得回退到旧候选问题主项解释；若 route 缺少对应问题说明，可按可见 outcome 的 `summary/firstAid/avoid/routeSafeSummary` 保守 fallback。

### 6.1 follow-up 与结果页消费约束（本轮同步）

`follow-up.vue` 与 `DiagnosePopup.vue` 应按 outcome 分组消费 `visibleOutcomes`，优先展示：
1) `actionAdviceItems`（该 outcome 的建议项）
2) `avoidAdviceItems`（该 outcome 的避免事项）

仅当该 outcome 无对应 per-outcome advice 时，才 fallback 到全局 `actionAdvice / nextSteps / whatToAvoid`，并避免“通用建议”重复拼接与重复展示。

QA 自动化会话 `diag_1779024952625_wjnp2qf9` 已在三类 outcome（`sunburn`、`fertilizer_repot_stress`、`overwatering_root_pressure`）下通过，结果页不再出现通用建议 fallback 或重复拼接。此行为作为当前 route-only 透传验收补充项。

建议长期收敛为：具体问题级建议、问题簇级建议、不确定原因级建议、非问题状态建议、重拍建议。不要把所有建议都硬绑定到具体问题，否则“不确定但方向明确”的场景会缺建议。

## 七、公开字段与内部字段边界

前端可展示中文标题、摘要、说明、建议、选项、公开问题 ID、公开选项 ID、视觉摘要、重拍建议、会话与轮次 ID。前端不应展示 `rankings`、`answerEffects`、`symptomClassRuntime` 私有结构、内部 `decisionCauseDetails` 全量对象、数据库表字段原名、模型原始输出。

### 7.1 route-aware 公共响应边界

1. `routeDecision` 是 debug/审计最小摘要，默认不作为前端通用稳定字段。当前净化后的实际字段为：`mode`、`visibleOutcomeKeys`、`activeRouteGroupKeys`、`nextQuestionKeys`、`fallbackPolicy`、`decisionCause`。不向用户端暴露 `primaryOutcomeKey/secondaryOutcomeKeys`、`routeTrace`、`gateResults` 等内部或旧兼容明细。
2. `visibleOutcomes`、`visibleOutcomeKeys`、`actionAdvice` 按 `route` 主链路输出，不再存在 ranking 排序、primary/secondary 分区或补位输出路径。
3. `diagnosis-round-presenter.js` 与 `buildFrontendDiagnosisResponse()` 必须保留 `actionAdvice`、`visibleOutcomes`、`outcomeMode`、`routeDecisionCause` 这些公开结果主链字段；前端归一化与结果页展示应优先消费结构化 `visibleOutcomes` 与 `actionAdvice`。旧历史中的 `primaryOutcome/secondaryOutcomes`、`primaryOutcomeKey/secondaryOutcomeKeys` 只能用于合成 `visibleOutcomes`，输出前删除。
4. `uncertain` 下不回填 `topProblem`/`topProblemKey`、`finalResult.primaryOutcome`/`primaryOutcome`，避免把 route 内部状态泄露到结果页。
5. `non_problematic` 不产生治疗型 `actionAdvice`，`uncertain` 不展示 `topProblem/topProblemKey`，仅给保守观察与补图建议。
6. `route` 模式下不向前端输出 `rankings` 或排名审计卡片；review/debug 只保留 route path、gate、候选 outcome 与证据说明。
7. `visual_discriminators` 与 `missing_info_for_path` 若存在，应进入 route debug 摘要与 presenter 归一化过程，作为可解释说明输入，而非直接替代最终业务结论。
8. 路由冲突不再作为清空可见 outcome 的规则；`visibleOutcomeKeys` 以 route 公开输出链为准，`maxVisibleOutcomes` 仅控制 `visibleOutcomeKeys` 全局显示条目上限。
9. 最终展示应以 `visibleOutcomes`（或其列表化映射 `outcomeItems`）为主入口，不得只取 `finalResult` 里的单条 `displayName`，也不得展示“核心结果/备选结果”两区。`finalResult` 仅保留旧兼容读取来源，不替代 `visibleOutcomes` 的多 outcome 展示链路。

### 7.2 `visibleOutcomes` 唯一公开出口

`visibleOutcomes` 是唯一用户可见 outcome 出口；多个 outcome 仍在同一 outcome 区域展示。前端诊断结果页、弹窗和 review 页面都应读取 `visibleOutcomes`，不展示“核心结果/备选结果”或 primary/secondary 分区。

旧历史数据若包含 `primaryOutcomeKey/secondaryOutcomeKeys` 或 `primaryOutcome/secondaryOutcomes`，只能作为合成兼容 `visibleOutcomes` 的输入；公共 API、compact/minimal response、review/detail 输出前必须删除旧字段。本轮验证 session `diag_1779084385746_rk42cja0`：`visibleOutcomes=[root_stress,nitrogen_deficiency,low_light_growth_weakness]`，`result/review/detail` 无 primary/secondary 输出。

## 八、验收口径与回放约束

- `diagnose-route-regression-runner` 已形成正式验收口径：三类真实样本云内回放 `3/3` 通过。
- 本地 `shell` 回放脚本保留为开发辅助，不作为 route 主链路完成的唯一判定条件；环境 DNS/fetch 波动导致的失败不直接等价于业务回归失败。

## 九、旧接口兼容

`toLegacyCompatiblePayload()` 和 `legacy-diagnose-presenter.js` 用于兼容旧诊断调用。新前端不建议继续依赖旧格式。

## 十、前端缺失源码情况下的接入建议

后续补前端源码后，应增加诊断首页上传图片流程、图片压缩与 `uploadCompression` 字段写入、诊断开始请求 payload、追问页状态机、结果页分支展示、历史与反馈页、错误态、重拍态、弱网态、SSE 进度展示等文档。

## 本轮输出收敛说明（2026-05-17）

- `公开响应 / routeDecision` 仅使用 `候选 outcome` 与 `路径 gate` 输出结果；禁止将 `ranking/rankNo/scoreDelta/topScoreGap/finalScore/baseScore` 作为前端展示依据。
- `answerEffects` 中的数值字段定义为“回答影响值”，仅用于解释候选 outcome 与答案证据关系，不得表述为加减分或名次变化。
- `review 前端` 明确不展示问题评分、分差或 ranking 卡片；允许展示 `候选 outcome`、规则命中证据、路径原因等可读中文字段。
