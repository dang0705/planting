# 09 植物识别、图片存储、天气与用户植物模块

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、植物识别服务

入口：`cloudfunctions/identify-http/app.js`。主要函数包括 `getAccessToken()`、`recognizePlant()`、`processBaiduResult()`、`classifyIdentityMatches()`、`pickPlantMatchFields()`。植物识别的定位是确定宿主上下文，不直接判断病虫害。识别不稳定时，诊断应降低对属级/宿主先验的依赖。

## 二、图片存储服务

入口：`cloudfunctions/storage-http/app.js`。主要能力包括上传诊断图片、上传用户植物图片、解析 dataURL 图片、构建云存储路径、生成临时访问 URL、删除诊断图片、查询植物图片列表。关键函数包括 `parseImageDataUrl()`、`resolveImageSuffix()`、`buildDiagnoseImageCloudPath()`、`buildPlantImageCloudPath()`、`uploadDiagnoseImage()`、`uploadPlantImageFile()`、`getFileTempUrl()`。

## 三、图片路径设计

图片路径应体现用户维度、植物维度、诊断维度和后缀安全。不能完全信任前端传入 suffix。

## 四、天气服务

入口：`cloudfunctions/weather-http/app.js`。主要函数包括 `fetchWeather(lat, lng)`、`formatWeatherData()`、`getNextMidnight()`、`getCachedWeather(openid)`、`saveCachedWeather(openid, weatherData)`。天气上下文用于辅助诊断，尤其是黄叶、萎蔫、低温、高温、湿度、通风相关问题，但不能替代用户养护事实。

## 五、用户植物服务

入口：`cloudfunctions/plant-user-http/app.js`，提供 `/user-plants` 相关能力。用户植物档案对诊断的价值包括植物身份锁定、属/科/类目补充、用户日常浇水/施肥/光照/通风习惯、历史图片和历史诊断关联。

## 六、植物目录服务

入口：`cloudfunctions/plant-catalog-http/app.js`。主要路由是 `/catalog/health`、`/catalog/map`、`/catalog/plants`。植物目录是诊断基础数据入口之一，应与 ``、植物 Excel 数据表、属级养护基线保持一致。

## 七、认证与用户身份

`auth-user-http` 负责用户资料和微信身份同步，主要函数包括 `syncWechatIdentity()`、`wechatLogin()`、`getUserById()`、`getUserByUnionId()`、`getUserByOpenid()`、`updateUserEmail()`、`updateUserPhoneNumber()`。`wechat-identity` 读取 CloudBase 注入的微信上下文；`wechat-phone` 解密微信手机号。

## 八、外围模块与诊断主链关系

| 模块 | 对诊断的影响 | 风险 |
|---|---|---|
| 植物识别 | 决定宿主先验和属级兼容 | 识别错会影响排序，但不应直接决定问题 |
| 图片存储 | 决定视觉模型输入质量 | 压缩过度、临时 URL 失效会影响视觉证据 |
| 天气 | 辅助环境判断 | 不能替代用户实际养护 |
| 用户植物 | 提供长期上下文 | 旧档案可能过期，需要允许本次上下文覆盖 |
| 认证 | 绑定用户数据 | openid 错误会造成数据串号 |
