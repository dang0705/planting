# 06 问题排序、证据计分与输出守卫

> 文档口径：本组文档基于 `Archive 2.zip` 中可见的 CloudBase 云函数代码、`rules.zip` 中的实施规范、以及现有粗文档 `diagnosis-runtime-code-logic.md` 整理。中文概念优先，英文函数名、字段名、路径只用于定位代码。
>
> 重要限制：压缩包内未发现 uni-app/小程序前端页面源码，因此前端实现只能按后端接口契约、返回结构和小程序接入规范反推，不写成“已确认源码行为”。

## 一、排序不是结论

当前系统会对候选问题排序，但排序最高不等于可以输出。必须经过输出资格、方向范围、上下文守卫和停止守卫。

## 二、候选问题池

候选问题池来自植物属/科/类目带来的先验、正式观察症状映射、诊断方向补充、用户回答直接影响、宿主兼容性兜底。如果候选池为空，系统必须输出不确定。

## 三、主要计分来源

- `computeVisualEvidenceScores()`：正式观察证据对候选问题的支持度。
- `computeQuestionEvidenceAndPenalty()`：用户答案对候选问题的支持、削弱和惩罚。
- `computeGenusFactor()`：属级兼容因子。它不能单独制造问题，只能在证据存在时调整排序。
- `computeHostFactor()`：宿主兼容因子。它不能替代症状证据。
- 直接调整：某些答案可直接调整候选问题，但必须有明确选项映射。

### 3.1 ranking 降级为候选与审计链

route 收口后，`ranking` 不再作为最终判定主链，改为：

1. 生成候选问题先验的优先级和回放顺序；
2. 在需要 fallback / 稳定性兜底时提供补位排序依据；
3. 与 `route`、`输出资格`、`停止守卫` 的联合判断一起决定是否可公开；
4. 不构成最终公开结论与行动建议的独立触发条件。

公开守卫必须继续以 `output-eligibility`、`必须追问 gate`、`uncertain` 合法性与 `non_problematic` 阻断为准，不得因 `top1` 排名高直接下发 final。

## 四、输出资格

`utils/output-eligibility.js` 和 `hasOutputEligibleProblemRanking()` 负责判断候选问题是否具备输出资格。输出资格至少关心：是否存在 active 正式观察证据、证据是否进入运行时、top 问题是否被当前症状模式允许、top 问题是否在当前诊断方向范围内、是否存在必要上下文、是否被负向问诊答案削弱、是否只是宽泛视觉异常支撑出来的候选。

## 五、上下文必需守卫

`utils/context-required-problem-guard.js` 处理“某些问题必须有额外上下文才能输出”的情况。例如只看到黄叶不能直接判断为缺铁/缺氮/缺水/积水；只看到孔洞不能直接判断为具体虫害；用户否定水肿鼓包后，不能直接跳到真菌叶斑，除非有独立叶斑证据。

## 六、诊断方向范围守卫

`scopeRankingsToDiagnosisDirections()` 会把输出候选限制在当前诊断方向内。如果最高分问题不属于当前已确认方向，应降低其公开输出优先级、输出不确定或继续追问补充方向证据。

## 七、宽泛视觉异常守卫

`distorted_growth`、`irregular_blotches`、`general_stress_mode`、`leaf_spot_complex_mode` 不能零问诊直接闭合为具体问题。如果没有可问高价值问题，应输出合法不确定。

## 八、黄叶守卫

黄叶类问题必须先分流：老叶/新叶、均匀/斑驳、是否伴随萎蔫、是否与浇水/光照/施肥/病虫害相关。追问后仍只有黄叶事实时，应输出不确定，而不是具体缺素/水分/光照问题。

整叶黄化（代码症状：`uniform_yellowing`）只能作为弱线索。它不能单独放行缺氮或营养不足，必须结合施肥、生长、新老叶分布或其他上下文证据后，才允许进入具体营养类结论。

“没有虫害痕迹”（代码选项：`no_pest_trace`）只能削弱虫害方向，不能反向给光照不足、晒伤或营养不足加分。用户否定某一方向时，系统只能降低该方向置信度，不能把这个否定答案当成另一个方向的正向证据。

蜜露/黏腻分支（代码症状：`sticky_honeydew`）也受黄叶虫害分流约束。系统不能在用户已经否定细网或蓟马银白擦伤后，又用泛化虫害类型维度（代码维度：`pest_trace_type`）把路径拉回同类虫害题；也不能在蜜露确认题回答“有”（代码问题与选项：`q_sticky_honeydew_confirm=yes`）后，再动态追问表面黏腻维度（代码维度：`surface_stickiness`）。如果用户确认有蜜露后已有表面黏腻感不存在（代码维度与选项：`surface_stickiness=no`）证据，不能闭合为红蜘蛛、蚜虫、白粉虱或蓟马等具体虫害，应输出不确定或等待更强证据。

黄叶虫害题的上下文守卫必须遵守分支边界。如果黄叶虫害痕迹类型（代码维度：`pest_trace_type`）已经回答“没有虫害痕迹”（代码选项：`no_pest_trace`），不能再强制蜜露确认题（代码问题：`q_sticky_honeydew_confirm`）。如果黄叶首要线索分流（代码维度：`yellowing_primary_clue_gate`）已经选择养护方向（代码选项：`care_context`）或病斑/霉层方向（代码选项：`disease_trace`），不能再强制细网确认题（代码问题：`q_spider_webbing_visible`）、蓟马银白擦伤确认题（代码问题：`q_thrips_silver_streaks`）、蜜露确认题（代码问题：`q_sticky_honeydew_confirm`）。

## 九、结构损伤守卫

孔洞、缺刻、叶片结构损伤必须区分虫咬、病斑脱落、机械旧伤、叶片展开损伤。追问后只有结构损伤事实，没有虫体、虫粪、啃咬边缘等正向证据时，不能直接输出具体虫害。

## 十、不确定合法性

常见合法原因包括 `input_unfillable`、`out_of_pool_no_mapping`、`insufficient_candidate_priors`、`no_observed_symptoms`、`broad_visual_differential_unresolved`、`context_required_problem_unconfirmed`。不确定是合法输出，不是失败。

## 十一、维护红线

不允许把排序第一直接当结论；不允许用属级/宿主先验替代症状证据；不允许用视觉候选替代正式观察证据；不允许在公开不确定结果里展示内部最高问题（代码字段：`topProblem`）；不允许因问题池耗尽而强行输出具体问题；不允许高特异规则绕过守卫；不允许用整叶黄化单独闭合缺氮/营养不足；不允许用“没有虫害痕迹”给光照或营养问题加分；不允许在蜜露确认后重复问同一黏腻维度；不允许在表面黏腻感被否定后闭合具体虫害；不允许养护或病斑/霉层分支被上下文守卫拉回虫害确认题。
