# 下一步开发重点 v1

## 结论

下一步最应该做的，不是继续改诊断页，而是**重构“加植物写链”**：

`目录选择 / AI 识别 -> plant-user-http -> user_plant_instances`

## 为什么是这一步

当前项目已经完成了这些关键收口：

- 目录读取侧已经切到新 identity / care 底座
- `identify-http` 已有最小正式视觉入口落库
- `diagnose-http` 已切到新 `route / outcome / snapshot` 主链
- 诊断前端承接层已基本补平

但“用户把一株植物真正加进自己的列表”这条写链，仍然停在旧模型。

## 证据

### 1. `plant-user-http` 的 POST 入口仍然只吃旧字段

见 [cloudfunctions/plant-user-http/app.js:47](/Users/jay/WebstormProjects/planting/cloudfunctions/plant-user-http/app.js#L47)：

- `plantId`
- `recognizedName`
- `sourceType`
- `recognitionType`
- `recognitionConfidence`
- `nickname`
- `location`
- `photos`

这里没有正式接入：

- `plantIdentityId`
- `identityResolutionStatus`
- `visualCallBatchId`
- `legacyPlantId`
- 识别候选确认后的 identity provenance

### 2. `createUserPlantInstance()` 仍按旧 `plant_catalog` 心智落库

见 [cloudfunctions/layer/utils/plant-knowledge.js:340](/Users/jay/WebstormProjects/planting/cloudfunctions/layer/utils/plant-knowledge.js#L340)：

它当前仍在写：

- `plant_id`
- `canonical_name`
- `recognized_name`
- `plant_genus`
- `plant_family_en`
- `plant_latin_name`

也就是仍然把“用户植物实例”建立在旧 catalog 字段镜像之上，而不是新 identity 中枢之上。

### 3. 前端加植物提交也还是旧 payload

见 [src/pages/add-plant/add-plant.vue:551](/Users/jay/WebstormProjects/planting/src/pages/add-plant/add-plant.vue#L551)：

前端提交时仍然只传：

- `plantId`
- `recognizedName`
- `nickname`
- `location`
- `photos`
- `sourceType`
- `recognitionType`

这意味着即使 `identify-http` 已经开始产出新 identity / visual runtime 信息，前端也没有把这部分信息带进用户植物实例落库。

## 不先做这一步的后果

如果现在跳过这条写链，继续做别的，会出现三个问题：

1. 新 taxonomy / identity 只停留在“读侧”和“诊断侧”，用户主数据入口仍是旧模型。
2. `identify-http` 的新视觉落库与 `user_plant_instances` 之间没有正式闭环。
3. 后面再回头改时，要同时返工：
   - add-plant 前端
   - `plant-user-http`
   - `createUserPlantInstance`
   - 用户植物列表字段解释

这会比现在直接补这条链更贵。

## 建议的实施顺序

### 第一步

改 `plant-user-http` POST 契约：

- 接受 `plantIdentityId`
- 接受 `identityResolutionStatus`
- 接受 `legacyPlantId`
- 接受 `visualCallBatchId`
- 接受识别来源与置信度的最小正式字段

### 第二步

改 `createUserPlantInstance()`：

- 以 `plantIdentityId` 为第一锚点
- `legacyPlantId` 仅作兼容映射
- 只保留第一版真正需要的镜像字段

### 第三步

改 `add-plant` 前端提交：

- 目录选择时提交 identity 锚点
- AI 识别确认时提交 identity / visual provenance
- 未命中时走受控 fallback

### 第四步

再补一次用户植物读取与详情富化的字段一致性检查，确认：

- 列表页
- 植物详情页
- 诊断入口

三处都以新 identity 语义为准。

## 一句话判断

**下一步最值的一刀，是把“用户植物实例创建”从旧 catalog 写法切到新 identity 写法。**
