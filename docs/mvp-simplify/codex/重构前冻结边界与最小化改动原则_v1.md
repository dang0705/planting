# 重构前冻结边界与最小化改动原则 v1

## 1. 文档目的

本文档用于在正式开始第一版重构前，先冻结边界，明确三件事：

1. 哪些模块明确不动。
2. 哪些模块只做最小适配。
3. 哪些模块属于本次重构中必须替换的业务中枢。

目标不是“尽量少改代码”本身，而是：

# **避免把业务中枢重构误做成基础设施层翻修。**

---

## 2. 总原则

### 2.1 第一版重构的本质

第一版要替换的是：

1. 植物身份中枢
2. AI 视觉入口中枢
3. 诊断会话与 outcome 中枢

第一版**不是**要重做：

1. 小程序启动方式
2. CloudBase 登录方式
3. HTTP 请求工厂
4. 通用上传组件
5. 天气、存储、认证等外围服务
6. 页面视觉设计体系

### 2.2 默认策略

默认采用以下策略：

1. **基础设施层默认不动**
2. **外围业务默认不动**
3. **界面壳默认保留**
4. **数据契约变化优先通过适配层吸收**
5. **只有旧业务中枢才允许重写**

### 2.3 一条执行纪律

# **禁止为了“顺手更干净”去改通用底座。**

如果某模块不是这次新框架的核心阻塞项，就不应在本轮一起翻修。

| 模块 | 文件 | 处理口径 | 原因 |

## 6. 第一版明确不做的事

以下内容在本轮重构中明确不做：

1. 不重做 UI 视觉设计。
2. 不重构认证体系。
3. 不改天气、存储、微信桥接等外围服务。
4. 不重写 HTTP 请求工厂。
5. 不重写 CloudBase 登录流程。
6. 不做 `species / identity / variant care overrides` 多层养护覆盖。
7. 不把 `visual_supervision_records` 做成训练闭环中枢。
8. 不在旧 legacy adapter 之上继续叠新语义。

---

## 7. 第一版推荐实施顺序

### 7.1 顺序

1. 先落新表与导入链。
2. 再重写植物身份与视觉入口中枢。
3. 再重写诊断中枢。
4. 最后做前端最小适配。

### 7.2 为什么这样排

因为当前最应该被保护的是：

- 前端壳
- 共用请求层
- 共用初始化层

而最应该被替换的是：

- 旧 `plant-knowledge` 中枢
- 旧 `identify-http`
- 旧 `diagnose-http`
- 旧 `plant-catalog-http`
- 旧 `plant-user-http`

这就是“重构 + 最小化改动”的真实含义：

# **只改业务心脏，不改血管和皮肤。**

---

## 8. 一句话

**第一版重构应采取“基础设施冻结、界面壳保留、外围服务不动、业务中枢替换、前端协议最小适配”的执行口径。除身份链、视觉入口链、诊断链和数据映射配置外，其余模块默认不进入本轮重构。**
