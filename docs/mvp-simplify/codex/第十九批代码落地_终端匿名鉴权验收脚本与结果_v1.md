# 第十九批代码落地: 终端匿名鉴权验收脚本与结果 v1

## 1. 本批目标

在不依赖微信小程序运行时的前提下，为当前项目补一条可重复执行的终端鉴权验收路径。

本批不改业务协议，也不改页面。

本批只解决两件事：

1. 终端是否能独立获得 CloudBase end-user token。
2. 获得 token 后，终端能把验收推进到哪一层。

---

## 2. 新增脚本

新增文件：

- `scripts/terminal-e2e/cloudbase-http-check.mjs`

这个脚本做两步：

1. 调用 `/auth/v1/signin/anonymously`
2. 带匿名 access token 调指定 HTTP 云函数路径

支持参数：

- `--env`
- `--path`
- `--method`
- `--query`
- `--body`
- `--openid`
- `--app-env`
- `--device-id`

设计原则：

1. 默认不打印 access token
2. 只输出 token 摘要、请求摘要和响应结果
3. 允许后续继续复用做 gateway-auth 层验收

同时在 `package.json` 新增了便捷入口：

- `npm run check:cloudbase-http -- --env=...`

---

## 3. 本批关键发现

### 3.1 终端已可独立获得匿名 access token

通过 CloudBase Auth HTTP API 实测确认：

- `POST /auth/v1/signin/anonymously`

在当前环境可直接成功返回：

- `access_token`
- `refresh_token`
- `scope = anonymous`

这意味着：

**“终端无法获得 end-user token”这一阻塞已被部分解除。**

### 3.2 之前的阻塞描述需要修正

此前阻塞文档里的说法应更新为：

不是“终端拿不到 token”，而是：

1. 终端拿不到微信小程序语义下的真实 `openid`
2. 当前业务函数依然把 `openid` 作为主用户锚点

也就是说，新的真实阻塞已经收敛成：

**token 可以拿到，但 token 不能自动映射出业务所需的 `openid`。**

---

## 4. 为什么匿名 token 还不足以完成真实业务验收

当前共用身份解析在：

- `cloudfunctions/layer/utils/http.js`

其逻辑是：

1. 优先信任 `x-openid / x-wx-openid`
2. 再尝试 `x-cloudbase-context`
3. 再尝试 runtime user info
4. 最后尝试从 bearer token 补查 profile

但当前业务层的正式入口：

- `plant-user-http`
- `diagnose-http`

仍然要求：

- `userInfo.openid` 必须存在

而匿名登录返回的是：

- `scope = anonymous`
- 一个可用的 `sub / access_token`

但不会自然给出微信 `openid`。

所以匿名 token 当前可以完成的是：

1. 网关鉴权层验证
2. HTTP 云函数安全规则验证
3. 废弃接口 / health 接口的受保护访问验证

还不能直接完成：

1. 添加植物
2. 诊断历史读取
3. 结果详情读取

因为这些链路仍然是 `openid`-first。

---

## 5. 本批实测结论

### 5.1 已确认成功

终端能够匿名登录 CloudBase Auth。

这说明当前环境的：

- `AnonymousLogin`

不仅是控制面配置为开启，而且数据面已经真实可用。

### 5.2 `diagnose-http/health` 已可通过匿名 token 访问

实测命令：

```bash
node scripts/terminal-e2e/cloudbase-http-check.mjs --env=cloud1-2grufevs395a9d5e --path=diagnose-http/health
```

结果：

1. 返回 `200`
2. `diagnose-http` 健康检查通过
3. 说明匿名 token 已足够穿过：
   - CloudBase Auth
   - 函数安全规则
   - HTTP 云函数网关

### 5.3 旧历史函数健康检查已进入废弃态

实测命令：

```bash
node scripts/terminal-e2e/cloudbase-http-check.mjs --env=cloud1-2grufevs395a9d5e --path=diagnosis-history-http/diagnosis/history/health
```

结果：

1. 返回 `200`
2. 响应体状态为 `deprecated`
3. 替代函数已明确指向 `diagnose-http`

这说明上一批对旧历史函数的“下线兼容态”已经真实生效。

### 5.4 业务历史接口仍因缺少 `openid` 无法完成正式验收

实测命令：

```bash
node scripts/terminal-e2e/cloudbase-http-check.mjs --env=cloud1-2grufevs395a9d5e --path=diagnose-http/diagnosis/history --query='{"page":1,"pageSize":1}'
```

终端直接看到的响应是：

1. 网关返回 `FUNCTION_EXECUTE_FAIL`
2. 平台请求号可用于反查云函数日志

继续用云函数日志复核后可确认：

1. 函数内部实际返回的是 `401`
2. 业务消息是 `请先登录`

这说明当前阻塞不是：

1. 函数没部署好
2. 网关 auth 没打通
3. 终端拿不到 token

而是：

**业务层仍要求 `resolveHttpUserInfo()` 最终产出 `openid`。**

### 5.5 当前剩余真实阻塞

终端还无法构造“与微信小程序用户身份严格绑定”的正式业务请求，原因不是 token，而是：

1. 终端没有微信运行时
2. 项目当前主业务仍以 `openid` 作为用户事实主键

所以后续如果要做严格意义上的“真实用户链路验收”，仍有两种路：

1. 在微信开发者工具里跑真实用户态
2. 终端补一个明确受控的测试身份映射方案

---

## 6. 当前价值

虽然本批还没有打通完整业务 E2E，但它已经把问题从“验收完全不可做”推进为：

1. 终端可以做 gateway-auth 层验收
2. 终端可以稳定复现匿名登录
3. 终端可以精确证明剩余阻塞在 `openid` 身份锚点，而不是 CloudBase 鉴权本身
4. 终端还能复核“旧函数已废弃、新函数健康可达”的云端现状

这对后续开发有两个直接价值：

1. 不必再反复怀疑 CloudBase Auth 能否在终端工作
2. 后续如果要做 identity 解耦或测试身份桥接，目标会更清楚

---

## 7. 下一步建议

下一步应在两条路中二选一：

1. 用微信开发者工具完成真正的 openid 绑定态 E2E 验收。
2. 在后端增加一条受控的 terminal-test 身份桥接，仅供开发环境验证主链。

如果继续坚持“不改当前 openid-first 业务主键”，那第一条更稳。

如果希望以后长期在终端做自动化验收，那第二条更有长期价值。
