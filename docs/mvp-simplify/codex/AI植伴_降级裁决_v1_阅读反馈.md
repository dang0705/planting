# AI植伴_降级裁决_v1 阅读反馈

## 1. 反馈目的

本文档用于在通读 [AI植伴_降级裁决_v1.docx](/Users/jay/WebstormProjects/planting/docs/AI植伴_降级裁决_v1.docx) 后，结合已有正式基线、`第一版开发现实可落地性裁决_v1_给Codex.md`、以及当前项目旧代码现状，给出我的执行层反馈。

这不是对原文档的否定，而是明确：

1. 哪些降级我认同并建议直接采用。
2. 哪些降级虽然方向对，但要补执行护栏。
3. 哪些点如果不修正，会在开发中产生隐性返工。

---

## 2. 总体判断

### 2.1 一句话结论

# **这份降级裁决是对当前新框架最有执行价值的一次收口。**

如果问我的直接态度：

# **我基本支持它作为第一版开发口径。**

而且相较于只谈“可落地性原则”的文档，这一版的优点在于：

1. 它把“理论可以成立”和“第一版到底怎么做”真正拉开了。
2. 它开始从一人维护、AI 工具协作、代码三个月后还能看懂这几个真实约束出发。
3. 它把很多原本容易继续膨胀的概念，压回到了能编码、能调试、能迭代的层级。

### 2.2 我的最终态度

我对这份降级裁决的评价不是“勉强可用”，而是：

# **它是目前最接近真实开发执行基线的版本。**

但这句话后面有一个限定：

# **建议采用，但要补两处修正，外加三组实现护栏。**

## 3. 我明确赞同的部分

### 3.1 四条红线设得对

文档提出的四条红线我基本全同意：

1. 植物身份链必须存在。
2. 诊断必须由 `route_primary_action` 驱动，不能退回旧的“症状直接出结论”。
3. 视觉调用必须有最小留痕。
4. outcome 必须支持三类，而不是只输出问题性结论。

这四条红线非常关键，因为它们刚好卡住了“降级不能把新框架降回旧系统”的边界。

### 3.2 9 阶段压成 3 段式调度，我支持

把 9 个阶段压成：

1. 输入接收与视觉解析
2. 身份解析 + 证据 + 路由
3. 诊断调度 + 结论输出

这是正确的降级。

原因不是因为 9 阶段的理论结构错了，而是：

1. 当前项目并没有成熟到需要完整阶段状态机。
2. 第一版的任务是跑通主链，不是先构建精密编排引擎。
3. 3 段式更适合当前 `identify-http` / `diagnose-http` 的改造路径。

我的看法是：

# **阶段概念保留在注释、日志、变量名里就够了，第一版不必把它们全做成状态机。**

### 3.3 运行时对象由 LLM 结构化 JSON 隐式承载，我支持

文档提出：

- `hypothesis_pool`
- `outcome_pool`
- 部分 evidence/hypothesis 竞争逻辑

第一版不建独立 SQL 表，而由 LLM 返回结构化 JSON 来承载，我认为这是正确的。

原因：

1. 这能显著降低 SQL 表爆炸风险。
2. 这和“第一版先跑通主链”的目标一致。
3. 只要代码层做强校验，这种方式是可控的。

这里的重点不是“让 LLM 随便输出”，而是：

# **让 LLM 承载候选结构，让代码负责校验边界。**

### 3.4 Taxonomy 只先落核心 2 张主表，我支持

第一版强制落：

- `plant_identity_entities`
- `plant_identity_aliases`

延迟：

- `plant_identity_match_rules`
- `plant_identity_merge_history`

这个取舍我同意。

理由很直接：

1. 没有 `entities + aliases`，新身份链根本起不来。
2. 但 `match_rules` 在第一批数据规模下完全可以先写在代码里。
3. `merge_history` 确实是后续治理对象，不该阻塞第一版。

### 3.5 视觉入口只强制 4 张核心表，我支持

我认同第一版必须落：

- `visual_call_batches`
- `visual_raw_image_records`
- `visual_normalized_image_results`
- `visual_admission_records`

并支持把：

- `visual_call_aggregate_results`
- `visual_supervision_records`

先做轻量或延后。

因为当前真正必须解决的是：

1. 调用了什么模型。
2. 原始输出是什么。
3. 标准化后变成了什么。
4. 系统为什么接纳 / 拒绝。

只要这 4 层在，第一版视觉链就已经具备最小可信追溯性。

### 3.6 问题题库、解释表、problem profile 延后，我支持

这份文档最有价值的一刀之一，是把以下对象从第一版硬依赖中移开：

- `question_templates / question_option_sets`
- `diagnosis_result_explanations`
- `plant_problem_profiles`
- `plant_identity_diagnosis_links`

我同意这里的主逻辑：

1. 当前真实数据量和调试节奏，并不支持先维护一整套静态题库。
2. 解释文案让 LLM 动态生成，比先维护 explanation 表更现实。
3. 先让问题性主链跑通，比先搭完所有静态知识层更重要。

### 3.7 追问与补图上限，我支持

这部分我基本完全同意：

- 最多追问 2 轮
- 最多补图 1 次
- 总轮次上限 3
- 每轮最多 2 个问题，每题最多 4 个选项

这不是产品保守，而是第一版系统可用性的必要条件。

如果不设这个上限，系统很容易退化成：

- 用户帮系统补标注
- 前端复杂度持续上升
- 后端会话逻辑迅速恶化

### 3.8 非问题性 outcome 白名单化，我支持

这部分判断是成熟的。

第一版非问题性只允许极少数白名单：

- `stable_natural_marking`
- `old_mechanical_damage`
- `normal_leaf_aging`
- `normal_new_growth`

这非常合理，因为：

1. 非问题性误判的风险其实不低。
2. 一旦 LLM 自由发挥“看起来正常”，第一版会快速失真。
3. 白名单化既保留了表达力，又控制了误判面。

## 4. 我认为必须修正的两处

### 4.1 `genus_care_profiles` 不能只按 `genus_name` 直接 join

这是我目前最明确的一处保留意见。

`AI植伴_降级裁决_v1.docx` 中写到：

- 第一版通过 `genus_name` 直接 join `genus_care_profiles`
- 不建 `plant_identity_diagnosis_links`

“不建挂接表”我可以接受。  
但是“只按 `genus_name` 直接 join”这个口径，我不建议直接采纳。

原因是当前正式基线已经明确冻结：

# **第一阶段 `genus_care_profiles` 的正式业务挂接键 = `genus_name + family_name_canonical`**

所以更合理的第一版执行口径应该是：

1. 可以不建 `plant_identity_diagnosis_links`。
2. 但代码里的业务 join 条件，仍应使用：
   - `genus_name`
   - `family_name_canonical`

也就是说：

# **可以降掉“挂接表”，但不要降掉“正式挂接键”。**

如果这里真的退回成只看 `genus_name`，会让第一版实现与已冻结的正式基线再次产生偏差，后面很容易返工。

### 4.2 新版 `diagnosis_sessions` 不应只是软约束，应该升为必需

文档把：

- `diagnosis_sessions（新版）`

列成了“建议落但允许简化”的软约束。

我对“允许简化”没意见，但对“只是软约束”有保留。

我的判断是：

# **新版轻量 `diagnosis_sessions` 应该升为第一版必需对象。**

原因很实际：

1. 追问轮次需要一个正式会话锚点。
2. 同一个 session 下会关联多个 `visual_call_batch`。
3. outcome 快照、当前主身份、当前 route、当前轮次都需要稳定归属点。
4. 历史记录与调试分析不能只靠零散 batch 和 raw 记录拼出来。

这不意味着要保留旧 `diagnosis_problem_rankings`、`diagnosis_symptom_observations` 那套重表结构。  
我主张的是：

- 建一个**轻量新版 `diagnosis_sessions`**
- 字段极简
- 允许用 JSON 快照承载部分运行时对象

但它应该是第一版核心表，而不是“有空再做”的软项。

## 5. 我认为需要补的三组实现护栏

### 5.1 prompt 可以承载决策，但代码必须做硬校验

文档把很多决策下放到 prompt 层，我整体支持。  
但我认为需要明确一条护栏：

# **prompt 负责生成，代码负责兜底裁决。**

至少以下内容不能只靠 prompt 自觉：

1. `route_primary_action` 只能是 4 个正式枚举之一。
2. `outcome_type` 只能是 3 类之一。
3. `non_problematic` 必须命中白名单，不允许自由发挥。
4. 追问题数不能超过 2。
5. 每题选项数不能超过 4。
6. retake / ask / final 的轮次条件必须被代码层复核。

也就是说，这份降级裁决在工程上成立的前提是：

# **LLM 输出必须经过 schema 校验和业务白名单校验。**

### 5.2 延后建表的对象，仍然要在代码结构里有独立模块

像这些第一版被延后的对象：

- `plant_identity_match_rules`
- `question_templates`
- `diagnosis_result_explanations`
- `plant_problem_profiles`

虽然可以先不建表，但我建议：

1. 匹配规则写在单独 matcher 模块里，不要散在 controller 里。
2. 追问生成、解释生成、outcome 正常化都要有独立 service / normalizer。
3. 常量和枚举必须集中定义。

这样后面即使把它们迁回正式表，也不会触发大面积重写。

### 5.3 旧链路与新链路必须做清晰隔离

这份降级裁决本质上已经是新系统的 V1 实施方案。  
因此我建议再补一条执行纪律：

# **不要继续在旧 `diagnose-http` 的 legacy adapter 层上堆新语义。**

原因是：

1. 旧代码当前已经带有适配层遗留。
2. 如果再把新 route / new outcome / new identity 继续叠进去，会让系统变成双语义混合。
3. 这会让 AI 工具后续也更难理解代码边界。

我的建议是：

- 要么同名重写，但把旧逻辑彻底剥离
- 要么直接按新模块重组

但不要再走“旧接口壳里不断叠新适配器”的路径。

## 6. 这份降级裁决最有价值的地方

### 6.1 它真正把“第一版成功标准”说清了

我最认可的一点，是文档不再追求“概念上全有”，而开始明确：

第一版成功不是因为对象齐全，而是因为用户能完成：

1. 上传图片
2. 身份解析
3. 诊断结果
4. 软建议

并且关键调用有最小追溯。

这正是一个一人维护项目应该追求的成功标准。

### 6.2 它把 AI 工具协作方式说清了

这份文档最贴近现实的一点，是明确把 AI 工具协作作为工程前提，而不是回避它。

尤其是这几条我认为很重要：

1. 每次任务先投喂降级裁决，再投喂模块正式基线，再投喂旧代码。
2. prompt 层承载复杂决策，代码层承载解析与校验。
3. prompt version 必须落库。
4. 代码注释、常量集中、概念对象变量名清晰。

这些要求不花哨，但非常有用。

## 7. 我对这份文档的最终采用建议

### 7.1 我的建议版本

如果让我拍板，我会这样采用它：

# **直接采用这份降级裁决作为第一版开发执行口径。**

但同时补两条修正：

1. `genus_care_profiles` 的业务挂接仍使用 `genus_name + family_name_canonical`，只是第一版不建独立挂接表。
2. 新版轻量 `diagnosis_sessions` 升为第一版必需表。

再补三条实现护栏：

1. 所有 LLM 输出都做代码级 schema + 白名单校验。
2. 所有延后建表对象都必须先在代码结构上模块化。
3. 新链路不得继续叠在旧 legacy adapter 逻辑上。

### 7.2 采用后的整体判断

如果按上面的修正版执行，我的判断是：

- 开发复杂度：明显可控
- 单人维护压力：可接受
- AI 工具协作效率：高
- 返工风险：中低
- 第一版跑通概率：明显高于原始完整版直接落地

---

## 8. 最终一句话

**我的看法是：这份降级裁决不是在削弱新框架，而是在把新框架压缩成一个真正能做完的第一版。它值得采用，但要补上“挂接键不能降”和“轻量 diagnosis_sessions 不能缺”这两处修正。**
