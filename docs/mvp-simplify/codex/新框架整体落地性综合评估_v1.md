# 新框架整体落地性综合评估 v1

## 1. 文档目的

本文档用于在阅读 `docs/第一版开发现实可落地性裁决_v1_给Codex.md` 后，结合当前仓库旧代码、CloudBase 现状、以及 A 类正式基线文档，对“新框架是否能在第一版真实落地”做一次综合评估。

这份文档不新增理论规范，主要回答四个问题：

1. 新框架的大方向是否成立。
2. 第一版按当前资源条件是否能真实落地。
3. 哪些部分可以直接做，哪些部分必须降级做。
4. 如果进入开发，应该按什么口径理解这套框架。

## 3. 综合结论

### 3.1 一句话结论

# **新框架整体方向成立，第一版具备真实落地性。**

但这个结论有一个前提：

# **必须按“现实可落地版本”实施，而不能按“理论最完整版本”硬做。**

如果遵守 `docs/第一版开发现实可落地性裁决_v1_给Codex.md` 的执行边界，这套框架对当前项目是：

- 方向正确
- 工程上可落地
- 单人维护下可控
- 可迭代扩展

如果不遵守，而是试图第一版一次性做满：

- 全量运行时对象精致化
- 全量 SQL 化
- 高成本稳定性采样
- 复杂人工审核流
- 复杂覆盖层与强状态机

那么整体落地性会明显下降，项目会迅速进入过度工程化。

### 3.2 本次最终判断

可以把本次综合评估压缩成一句执行口径：

# **新框架不是不能做，而是必须先做“最小真系统”，不能先做“最大全景系统”。**

## 4. 为什么我判断它“可落地”

### 4.1 框架大方向没有根本性错误

从目前正式基线与“现实可落地性裁决”对照来看，没有发现需要整套推翻的根本错误。以下方向是成立的：

1. Taxonomy、Diagnosis、Runtime 分层成立。
2. 身份链与症状链并行成立。
3. route hint 不反写事实层成立。
4. `observed_evidence_set -> route_primary_action -> diagnosis/outcome` 的主链成立。
5. 属级养护基线放在 Taxonomy 侧成立。
6. 问题性 / 非问题性 / 不确定三类 outcome 分层成立。

也就是说，现在不是“框架本身错了”，而是“第一版不能把这套框架实现得过满”。

### 4.2 正式基线已经完成第一轮工程收口

经过前几轮文档复核，当前最关键的工程阻断项已经被收口到可开发状态，例如：

1. 正式基线文件清单已经明确。
2. `identity_resolution_status` 已经从 Taxonomy 静态层剥离。
3. `genus_care_profiles` 的第一阶段挂接键已固定为 `genus_name + family_name_canonical`。
4. `visual_call_batches` 已正式进入第一版 SQL 基线。
5. `route_primary_action` 正式枚举已冻结。
6. `hold_for_review` 已删除，不再引入人工审核状态机。
7. 最小 SQL 化策略已经明确，不再要求所有运行时对象都建表。

这意味着新框架不是“还在概念漂浮阶段”，而是已经具备落代码的最小约束。

### 4.3 当前项目并不是白纸，已有可复用骨架

从代码现状看，虽然旧业务模型需要重构，但工程骨架并不缺：

1. 前端小程序壳已存在。
2. 首页、植物详情、诊断弹窗、添加植物页的交互框架已存在。
3. HTTP 云函数调用方式已存在。
4. `identify-http`、`diagnose-http`、`plant-user-http` 这些边界已经成型。
5. 数据导入 / diff / publish / rollback 的 CLI 骨架已经存在。

这非常关键，因为它把落地问题从“从零搭系统”降低成了“替换旧中枢”。

### 4.4 CloudBase 基础设施本身不是阻断项

基于最近两天的 MCP 核查：

- CloudBase 环境为 `NORMAL`
- MySQL 可用
- 云函数在线
- 存储与静态托管存在

所以第一版落地的阻力主要不在云基础设施，而在：

1. 新旧 schema 的迁移
2. 新旧接口契约的切换
3. 对第一版复杂度的控制

## 5. 为什么它“不能按理想完整版硬做”

### 5.1 当前模型能力不支持做“终局裁判”

`docs/第一版开发现实可落地性裁决_v1_给Codex.md` 最关键的一条是：

# **模型不是裁判，是候选生成器。**

这条判断非常准确，而且和现有项目现实完全一致。

原因是：

1. 百度识别返回的是名称候选，不是内部 identity 真值。
2. 混元视觉可以做粗判、候选、弱提示，但不适合在第一版承担高置信最终裁决。
3. 如果把模型结果直接提升为系统真值，会立即产生伪精确。

所以第一版里：

- 百度只能做 `raw_recognition_name` 输入源
- 混元只能做候选提取、粗筛和路由弱建议
- 系统本身负责接纳、分层、裁决

这不是保守，而是让框架可落地的必要前提。

### 5.2 用户交互成本必须被严格限制

这份现实裁决里另一条非常关键：

# **用户不是标注员。**

这和我对当前产品壳的判断完全一致。

当前项目的真实入口是首页植物卡片和诊断弹窗，不是复杂工作台。  
这意味着第一版交互必须符合低配合度用户预期：

1. 最多 1 次补图。
2. 最多 2 轮追问。
3. 优先高收益问题。
4. 不能把补图变成默认主路径。

如果不限制，产品会从“用户做一次诊断”变成“用户替系统补标注数据”，这会直接拉低真实可用性。

### 5.3 运行时对象不能第一版全部精致化

当前新文档体系里，运行时对象概念已经比较完整，这本身不是问题。  
问题在于第一版如果把这些对象全部做成：

- 独立 SQL 表
- 独立状态机
- 独立治理策略
- 独立可回放链路

那么一人维护几乎不可控。

因此，“现实可落地性裁决”提出的最小 SQL 化策略是正确的：

#### 第一版应强制 SQL 化

- `visual_call_batches`
- `plant_identity_resolution_records`
- `visual_raw_image_records`
- `visual_normalized_image_results`
- `visual_admission_records`

#### 第一版可先不强制 SQL 化

- `normalized_input`
- `observed_evidence_set`
- `hypothesis_pool`
- `outcome_pool`
- `question_queue`
- `stop_state`
- `diagnostic_trace`

这不是降低框架质量，而是把框架从“理论系统”压成“可运行系统”。

## 6. 新框架各模块的落地性判断

### 6.1 Taxonomy 静态层

**落地性判断：高**

原因：

1. 静态主数据边界已经收口。
2. 关键表与挂接键已经明确。
3. 该部分主要是数据建模与导入问题，不是高不确定运行时问题。

落地建议：

- 直接按正式 SQL 基线与导入方案实现。
- 不再回退到旧 `plant_catalog` 作为身份真相中心。

### 6.2 植物身份解析层

**落地性判断：中高**

原因：

1. `matched / weak_matched / unresolved` 这套状态现实且有效。
2. `taxonomy_match_status` 与 `identity_resolution_status` 的边界已经明确。
3. 当前旧项目虽没有这套模型，但有旧匹配逻辑可作为迁移参照。

风险点：

1. 容易被做成过重状态机。
2. 容易把百度返回名误当 identity 真值。

执行口径：

- 第一版把身份解析理解为“候选进入 + Taxonomy 裁决 + 会话内主身份确定”。
- 不围绕状态做复杂跳转图。

### 6.3 AI 视觉入口层

**落地性判断：中高**

原因：

1. 新文档已明确 `visual_call_batches` 与相关对象。
2. 旧项目已有图片上传与 `identify-http` 壳，可承接改造。
3. 这层在第一版里可以被解释为“候选输入链 + 接纳链”，不必做成全自动真值层。

风险点：

1. 容易因为追求“稳定性分数”而引入多次模型调用。
2. 容易因为对象过多而做成治理平台。

执行口径：

- `top1_stability_score` / `top3_stability_score` 字段可保留，但允许留空或弱估计。
- 第一版禁止为了算稳定性而做重复调用采样。

### 6.4 诊断运行时主链

**落地性判断：中**

这是整个系统最重的一块。

原因：

1. 旧项目当前的诊断中心仍是旧 `symptom/problem/followup` 体系。
2. 新框架要求先形成 `observed_evidence_set`，再形成 `route_primary_action`，再进入 outcome。
3. 这意味着 `diagnose-http` 需要重写核心调度逻辑。

但是它仍然可落地，前提是：

1. 不把全部 8 大运行时对象都做成正式持久化对象。
2. 不做复杂回放与纠错状态机。
3. 不做高成本模型重复调用。
4. 诊断目标先聚焦问题性结论与不确定结论。

### 6.5 追问与补图机制

**落地性判断：中高**

这部分的落地关键不在于技术，而在于约束。

如果按现实裁决执行：

- 最多 1 次补图
- 最多 2 轮追问
- 优先高收益问题

那么这部分是可落地的。

如果不限制：

- 会迅速膨胀为复杂问卷流
- 同时拖垮前端体验和后端状态复杂度

### 6.6 Outcome 结论层

**落地性判断：中高**

执行策略应为：

1. 问题性结论作为第一版主做对象。
2. 不确定结论必须保留。
3. 非问题性结论白名单化。

这是非常现实的收口方式，因为它既保住了系统表达力，也避免了第一版在“正常/异常边界”上过度冒险。

### 6.7 属级养护基线与动作建议

**落地性判断：高**

但有明确前提：

- `genus_care_profiles` 可以落地
- 养护动作建议必须软化

这意味着第一版建议只能输出：

- 更偏向
- 通常建议
- 当前优先检查
- 可先尝试调整

不能输出硬处方型指令。

这和属级养护基线本身的能力边界是一致的，也是对第一版风险最小的做法。

### 6.8 监督与治理层

**落地性判断：中**

这块第一版能做，但只能轻做。

`visual_supervision_records` 在第一版里应理解为：

# **留痕表**

而不是：

- 自动训练闭环中枢
- 全量质量治理平台
- 自优化控制台

如果只是留痕，它可落地。  
如果想第一版做闭环治理平台，它不可取。

## 7. 与当前旧项目结合后的真实难点

### 7.1 真正的工程难点不是“文档还不够”

现在文档最难的阶段已经过去。  
真正的难点已经转移到代码与数据迁移层：

1. 旧 `plant_catalog` 中心要迁出。
2. 旧 `identify_sessions` 语义要退场。
3. 旧 `diagnosis_sessions` 中枢要退到兼容层或历史层。
4. 前端请求/响应对象要整体替换。

所以后续开发的本质是：

# **旧系统替换工程**

而不是：

# **继续补文档工程**

### 7.2 第一版最大的失败风险来自“做太多”

从综合评估看，第一版最容易失败的原因不是“框架太先进”，而是：

1. 同时重写太多层。
2. 把运行时对象全表化。
3. 把模型当裁判。
4. 把补图和追问拉太长。
5. 把非问题性结论做太细。
6. 把养护建议输出成硬处方。

也就是说，最大风险不是方向错误，而是执行失控。

### 7.3 第一版最大的成功条件是“边界纪律”

当前框架能否落地，不取决于文档厚度，而取决于以下纪律是否被严格遵守：

1. 只做第一版必需的 SQL 表。
2. 运行时对象轻量化。
3. 模型统一降级为候选生成器。
4. 严格限制交互次数。
5. Outcome 先收口到问题性 + 不确定 + 少量白名单非问题性。
6. 养护建议只做软建议。

## 8. 第一版的推荐落地口径

### 8.1 正确理解方式

第一版应把新框架理解成：

# **一个分层清楚、边界清楚、允许降级实现的业务框架**

而不是：

# **一个要求第一天就把所有对象做满的理想状态机系统**

### 8.2 推荐落地策略

推荐按以下顺序实施：

1. 新 SQL 表结构与导入脚本先落地。
2. 新 Taxonomy / identity 查询链落地。
3. 新视觉入口链落地。
4. 新诊断调度链落地。
5. 前端协议适配。
6. 旧链路逐步退出核心地位。

### 8.3 第一版“算成功”的标准

不是功能全满，而是满足以下标准：

1. 用户能完成一次从上传图片到获得结果的完整闭环。
2. 系统能完成植物身份候选、Taxonomy 裁定与会话主身份解析。
3. 系统能建立正式接纳后的证据集。
4. 系统能基于 `route_primary_action` 决定下一步。
5. 系统能输出问题性结果、不确定结果、或少量白名单非问题性结果。
6. 系统能输出软建议而不是硬处方。
7. 关键调用与解析过程有最小可追溯记录。

只要做到这些，第一版就是成功的。

---

## 9. 最终裁决

### 9.1 总体结论

# **新框架整体可落地。**

但更精确的说法应当是：

# **新框架在“现实可落地裁决”约束下，具备中高落地性。**

### 9.2 具体判断

- 框架方向：高可信，可继续推进
- 工程实现：可推进，但属于结构化替换，不是小修小补
- 单人维护：可控，但前提是严格降级
- 第一版上线可行性：可行，但必须克制复杂度
- 理想完整版一次性实现：不建议

### 9.3 最重要的一句话

**这套新框架不是要被“削弱”，而是要被“正确解释后落地”。第一版真正应该实现的，是它的最小真实版本，而不是它的最大理论版本。**
