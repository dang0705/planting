# genus_care_profiles 建表立场补充说明

## 1. 结论

# **我认同现在就建 `genus_care_profiles` 表。**

而且我认同你这句话的核心判断：

# **`genus_care_profiles` 的建表本身，不会显著抬高第一版整体难度。**

相反，如果第一版为了“再降一点复杂度”而不建这张表，后面大概率会引入更大的结构性返工。

---

## 2. 为什么我认同现在建表

### 2.1 这张表本来就在正式基线里，不是额外加戏

当前正式基线已经明确把它纳入了第一版范围：

- `docs/new-rules/最终_SQL_制表方案_v1_3_完整最终版_开发收口版.md`
- `docs/new-rules/第一版_SQL_表结构草案_v1_1_开发收口版.md`
- `docs/第一版开发现实可落地性裁决_v1_给Codex.md`

所以现在建 `genus_care_profiles`，不是给第一版新增任务，而是把已经冻结的任务按正确方式落地。

### 2.2 当前线上已经有旧版原型，迁移成本不是从零开始

仓库和线上当前都已经存在旧底座：

- 旧表：`genus_care_profile`
- 旧代码挂接：`cloudfunctions/layer/utils/plant-knowledge.js`

这说明：

1. 数据原型已经有了。
2. 旧查询模式已经有了。
3. 当前增加的主要工作不是“发明一套新能力”，而是“把旧属级养护底座迁到新正式模型”。

换句话说，这不是高不确定新模块，而是中等复杂度的结构迁移。

### 2.3 不建表会让后续代码结构出现假接口

如果第一版不建 `genus_care_profiles`，但又想保留属级养护能力，代码里通常会出现三种临时替代：

1. 把养护 JSON 临时塞回 `plant_identity_entities`
2. 把 prior / care 数据塞进 prompt 常量或本地配置
3. 继续依赖旧 `genus_care_profile` 表并在新链路里做兼容拼接

这三种方式短期都能“先跑”，但都会在第二阶段引入结构性返工：

- 数据责任边界重写
- join 逻辑重写
- repository 重写
- 导入链重写
- prompt context 组装逻辑重写

所以从工程角度看：

# **不建表不是省工作，而是把工作推迟成更贵的重构。**

### 2.4 这张表会稳定影响多个模块，不是孤立对象

`genus_care_profiles` 不只是“给建议文案用一下”的边缘对象，它会稳定影响：

1. Taxonomy 侧宿主背景
2. 软建议生成
3. prior / diagnosis context 组装
4. 前端植物详情的养护展示
5. 后续问题画像与建议层扩展

既然它会穿过这么多模块，那么第一版就把它放到正式位置上，反而能减少后续接口漂移。

---

## 3. 我认同的不是“全养护体系全开”，而是“先把主表建对”

这里要区分两个概念：

### 3.1 我支持

- 现在就建 `genus_care_profiles`
- 现在就把导入链和查询链接到这张表
- 现在就让它成为第一版软建议和宿主背景的正式来源

### 3.2 我不支持

- 第一版同时做 `species care overrides`
- 第一版同时做 `identity care overrides`
- 第一版同时做 `horticultural variant overrides`
- 第一版同时做复杂的 care 多层覆盖继承系统

也就是说，我的立场是：

# **主表现在就建。覆盖层以后再说。**

这两件事不能混在一起判断复杂度。

---

## 4. 真正需要控制复杂度的地方

在 `genus_care_profiles` 这件事上，真正会抬高复杂度的，不是建表本身，而是以下三件事：

### 4.1 把建议输出做成硬处方

第一版如果只是输出软建议：

- 更偏向
- 可先尝试
- 优先检查
- 建议结合当前环境调整

那复杂度是可控的。

真正危险的是把它做成：

- 必须几天浇一次
- 一定是某项养护失误
- 直接给强处方

所以复杂度控制点在“输出口径”，不在“要不要建表”。

### 4.2 挂接键做错

第一版真正不能退让的，是业务挂接键。

当前正式基线已经冻结：

# **`genus_care_profiles` 的第一阶段正式业务挂接键 = `genus_name + family_name_canonical`**

所以可以不建独立挂接表，但不能把 join 逻辑降成只看 `genus_name`。

### 4.3 顺手把它扩成复杂治理系统

第一版只需要：

1. 表结构稳定
2. 数据能导入
3. 查询能使用
4. 软建议能生成

不需要第一天就做：

- 覆盖链
- 版本治理 UI
- 自动冲突修正
- 审核工作流

---

## 5. 我的最终立场

如果把问题问得更精确一点：

### 问：第一版现在就建 `genus_care_profiles`，值不值得？

答：

# **值得。**

### 问：现在不建，是否会降低第一版难度？

答：

# **表面上略降，实际上会把后续重构成本抬高。**

### 问：现在建，会不会把第一版拖进过度工程化？

答：

# **不会，只要你建的是“主表”，不是“全养护覆盖系统”。**

---

## 6. 一句话

**我的明确态度是：`genus_care_profiles` 应该现在就建，因为它属于低到中复杂度的基础建模工作，而不是高复杂度的运行时治理系统；现在不建，后面反而更容易引发结构性返工。**
