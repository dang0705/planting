# 第十八批代码落地: 旧诊断历史函数下线与真实登录态验收阻塞说明 v1

## 1. 本批目标

本批不继续扩诊断主链功能，而是先处理一个更危险的工程问题：

1. 旧 `diagnosis-history-http` 仍然存在，理论上还能返回另一套诊断历史结果。
2. 真实登录态端到端验收仍未打通，需要把阻塞点写清楚，避免后续反复误判。

本批的核心原则是：

**先收口公开接口口径，再继续做验收。**

---

## 2. 旧历史函数为什么必须先下线

当前前端实际已经改走：

- `diagnose-http/diagnosis/history`
- `diagnose-http/diagnosis/result`
- `diagnose-http/diagnosis/feedback`

但仓库里仍保留：

- `cloudfunctions/diagnosis-history-http/app.js`

这意味着仓库语义上还存在两条历史读取链：

1. 新链：`diagnose-http`
2. 旧链：`diagnosis-history-http`

虽然当前前端不再直接使用旧函数，但旧函数仍可能被：

1. 后续误调用
2. 外部脚本误调用
3. 维护时误认为“仍然是正式接口”

这会带来两个问题：

1. 同一业务存在两套输出口径
2. 后续排查历史/结果问题时会出现“看的是哪条链”的歧义

---

## 3. 本批代码改动

### 3.1 `diagnosis-history-http` 收成废弃兼容函数

修改文件：

- `cloudfunctions/diagnosis-history-http/app.js`

现在它的行为变成：

1. 仅保留 `/diagnosis/history/health`
2. `/diagnosis/history`
3. `/diagnosis/history/detail`
4. `/diagnosis/history/feedback`
5. `/diagnosis/decision`

以上旧业务路径统一返回：

- HTTP `410`
- 明确提示迁移到 `diagnose-http`

返回体中会写明推荐替代路径：

- `diagnose-http/diagnosis/history`
- `diagnose-http/diagnosis/result`
- `diagnose-http/diagnosis/feedback`

### 3.2 部署描述同步收口

修改文件：

- `cloudfunctions/diagnosis-history-http/cloudbase-functions.json`

处理方式：

1. 移除旧 history/detail/feedback 路由声明
2. 只保留 health 路由

这个动作的目的不是依赖本文件“自动删云端路由”，而是先把仓库内的部署语义收口为：

**旧函数不再承担正式历史接口职责。**

---

## 4. 为什么这一步优先于真实登录态验收

因为当前真实登录态验收还存在平台鉴权阻塞，如果先不把旧链收口，后面即便勉强打通验收，也仍会留下“历史到底该看哪个函数”的问题。

所以本批先做：

1. 统一口径
2. 下线旧函数业务能力
3. 保留 health 作为兼容可见性

这样后续所有诊断相关验收都只应围绕：

- `diagnose-http`

展开。

---

## 5. 当前真实登录态验收的实际阻塞

### 5.1 已确认的前提

当前 CloudBase 环境：

- `cloud1-2grufevs395a9d5e`
- 状态：`NORMAL`

登录策略已确认：

- `AnonymousLogin = true`
- `EmailLogin = true`
- `UserNameLogin = true`

也就是说，平台侧不是“完全没开登录能力”。

### 5.2 真正的阻塞点

真实登录态端到端验收当前卡在两层：

1. 相关 HTTP 云函数安全规则要求 `auth != null`
2. 本项目真实取 token 的链路依赖小程序运行时：
   - `wx.cloud`
   - `@cloudbase/js-sdk`
   - 小程序登录态

终端侧当前没有直接复用这条链路的现成环境。

### 5.3 已核实的函数约束

以下函数都不是匿名终端请求可直接验收的：

- `auth-user-http`
- `diagnose-http`

原因不是业务代码拒绝，而是：

- 网关安全规则先要求用户态鉴权成立

### 5.4 为什么还没直接打通匿名登录验收

目前已确认：

1. 前端真实 token 获取依赖小程序 CloudBase 登录流程
2. 终端环境没有可直接复用的 miniapp runtime
3. 本仓库当前也没有一条已经落好的“纯 Node 端 end-user token 获取脚本”

因此，真实登录态验收不是业务代码阻塞，而是：

**终端侧缺少一条可复用的 end-user 鉴权取得路径。**

---

## 6. 当前结论

本批之后，诊断相关公开链路的工程口径变成：

1. 正式诊断入口：`diagnose-http`
2. 正式诊断历史：`diagnose-http/diagnosis/history`
3. 正式结果详情：`diagnose-http/diagnosis/result`
4. 正式反馈入口：`diagnose-http/diagnosis/feedback`
5. `diagnosis-history-http` 仅作为废弃兼容函数保留 health 可见性

这一步的价值在于：

**先把“哪条链是正式链”彻底说清楚。**

---

## 7. 下一步建议

下一步应继续做两件事：

1. 为终端补一条可复用的 end-user 鉴权获取路径
2. 在此基础上完成一次真实登录态下的：
   - 添加植物
   - 发起诊断
   - 查看历史
   - 查看结果详情

在这两步完成前，不应再把 `diagnosis-history-http` 当成正式接口补丁点继续扩写。
