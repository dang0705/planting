# 第十三批代码落地：诊断前端契约兼容收口 v1

## 1. 背景

在第九至第十二批改造后，`diagnose-http` 已经正式输出：

- `outcomeType`
- `routePrimaryAction`
- `identityResolutionStatus`
- `stopReason`
- `confidenceLevel`

但前端承接层仍然带有旧问题性诊断心智，主要有三类偏差：

1. `uncertain / non_problematic` 会被压扁成普通问题性结果。
2. 本地诊断历史仍用 `Date.now()` 作为临时 ID，无法稳定对应真实诊断会话。
3. 诊断结果承接页在本地回退和远端回放时，对 `sessionId / resultId` 的命中不稳定。

本批目标不是重做页面，而是在不改视觉结构的前提下，把承接层契约补平。

## 2. 代码改动

### 2.1 `src/utils/diagnose-flow.js`

诊断结果整形逻辑改为显式识别新版结果语义：

- 新增 `outcomeType` 归一化。
- 新增 `non_problematic / uncertain / followup` 对应的健康状态文案。
- 新增 `routePrimaryAction / identityResolutionStatus / stopReason / confidenceLevel / confidenceReasons / needHumanReview` 保留。
- `mainIssueText / summaryText` 不再只按“问题性结论”推断，改为按 `outcomeType + summaryCard + explanation` 联合生成。
- `buildFollowUpPayload()` 统一输出新版 `answers[{ questionId, optionId }]` 结构。

### 2.2 `src/http-functions/diagnose/client.js`

补强结果读取侧的兼容整形：

- 对已经是新版结构的结果对象补默认字段，而不是直接原样透传。
- 保证 `requestDiagnosisResult()` 返回的 detail 至少带：
  - `resultId`
  - `diagnosisSessionId`
  - `stage`
  - `status`
  - `outcomeType`
  - `routePrimaryAction`
  - `identityResolutionStatus`

这样远端历史详情和实时诊断结果会落到同一前端心智模型。

### 2.3 `src/store/diagnose.js`

本地诊断历史从“临时时间戳 ID”切成“稳定诊断 ID”：

- 优先使用 `diagnosisId / diagnosisSessionId / resultId`
- 插入历史前先按稳定 ID 去重

这样再次进入结果页时，不会因为本地 store 只存临时 ID 而丢失命中。

### 2.4 `src/pages/diagnose/diagnose.vue`

只读结果页补了两处脚本层兼容：

- 本地回退命中时，同时支持：
  - `id`
  - `diagnosisId`
  - `diagnosisSessionId`
  - `resultId`
- 远端结果展示时不再硬编码 `stage = final`，改为优先使用接口返回阶段。

## 3. 校验

已完成静态校验：

- `node --check src/utils/diagnose-flow.js`
- `node --check src/http-functions/diagnose/client.js`
- `node --check src/store/diagnose.js`

`src/pages/diagnose/diagnose.vue` 本批只改了 `<script setup>` 中的计算逻辑，没有改模板结构。

## 5. 影响判断

本批完成后，前端承接层已经不再把新版诊断框架强行压回旧问题性模型，至少在以下场景下会更稳：

- `uncertain` 结果回显
- `non_problematic` 结果回显
- follow-up 轮次后的本地历史更新
- 结果页通过 `sessionId / resultId` 回放远端结果

这批属于“契约补平”，不是新 UI 开发，也不是诊断页面重做。
