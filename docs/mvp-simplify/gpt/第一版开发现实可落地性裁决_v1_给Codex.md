# 第一版开发现实可落地性裁决 v1（给 Codex 的实现约束文档）

> 适用对象：
>
> - Codex
> - 第一版 dev 开发阶段
>
> 目标：
>
> - 不是继续抬高系统理论天花板
> - 而是从**真实可落地、可维护、一人可控**的角度
> - 对当前正式基线文档中的规范做一次执行层裁决
>
> 本文档用于回答一个核心问题：
>
> # **哪些概念和规范现在可以直接实现**
> # **哪些概念虽然正确，但第一版必须降级**
> # **哪些概念第一版绝不能做成硬规范**
>
> 使用方式：
>
> - 当前正式基线仍然有效
> - 但一旦正式基线中的某条规范与本文档发生“执行层冲突”，
> - **第一版开发阶段以本文档为执行优先级更高的现实裁决**
>
> 核心原则：
>
> # **先把第一版做成可跑通、可验证、可迭代的系统**
> # **不要把第一版做成概念完美但无法维护的系统**

---

# 一、总裁决

当前正式基线中，没有发现“大方向完全错误”的核心规范。  
也就是说：

- 身份链与症状链并行
- 接纳判定分层
- 结论层分为问题性 / 非问题性 / 不确定
- 属级养护基线放在 Taxonomy 侧
- route hint 不反写事实层
- Taxonomy / Diagnosis / Runtime 分层

这些方向总体都成立。

但是，当前基线中存在一批：

# **概念上正确**
但
# **第一版如果硬按理想版本实现，会显著增加失败率**

的问题。

因此，第一版开发必须统一采用：

# **现实可落地版本**
而不是
# **理论最完整版本**

---

# 二、第一版开发的最高现实原则

## 2.1 模型不是裁判，是候选生成器
第一版里：

- 百度植物识别
- 混元视觉

都不能被实现成“最终真值裁判”。

### 正式裁决
第一版必须统一理解为：

# **模型负责生成候选**
# **系统负责做接纳、分层、裁决**

---

## 2.2 用户不是标注员
第一版不要假设用户会：

- 连续补很多张图
- 回答很多轮追问
- 理解复杂植物术语
- 主动配合高复杂诊断流程

### 正式裁决
第一版所有交互设计都要以：

# **低配合度用户**
为默认前提。

---

## 2.3 一人维护优先于理论完整
凡是会引入以下后果的设计，都不应在第一版硬做：

- 状态机过重
- 运行时对象过多
- SQL 表爆炸
- 日志与回溯链太复杂
- 模型重复调用成本过高
- 需要持续人工审核 / 标注 / 清洗

### 正式裁决
第一版优先级顺序为：

1. 可运行
2. 可验证
3. 可维护
4. 可扩展
5. 理论完整

而不是反过来。

---

# 三、模型能力现实边界裁决

## 3.1 百度植物识别的第一版职责

### 允许承担
- 植物名称候选输入
- Taxonomy 匹配触发源
- 命名归一前置输入

### 不允许承担
- 植物身份真值机
- genus / family / species 最终裁定
- 症状识别
- 病理判断
- 诊断排序依据

### 实现要求
百度返回结果在第一版中只能进入：

- `raw_recognition_name`
- Taxonomy 匹配逻辑

不得直接跳过 Taxonomy 匹配进入：

- `matched_plant_identity_id`

### 原因
百度返回的是：

- 名称候选
- 近似命名
- 商业名 / 俗名混杂输入

它不是你内部的 identity 主表。

---

## 3.2 混元模型的第一版职责

### 允许稳定承担
1. 图像质量粗判  
2. 器官 / 部位弱提示  
3. 症状候选提取

### 允许弱承担
4. 模式候选  
5. 路由建议

### 第一版不允许强依赖
6. 多次调用稳定性采样  
7. 高置信最终诊断裁决  
8. 复杂模式识别作为硬规则入口  
9. 症状真值自动锁定

### 正式裁决
第一版必须统一理解为：

# **混元 = 候选生成器 + 粗筛器**
不是：
# **植物症状专用稳定裁判**

---

## 3.3 稳定性分数的现实裁决

当前正式基线里有：

- `top1_stability_score`
- `top3_stability_score`

### 第一版裁决
这些字段可以保留，但：

# **第一版不得为了计算它们而强制多次重复调用模型**

### 允许做法
- 单次调用内部弱估计
- 留空
- 后续版本补

### 不允许做法
- 同图跑 3 次 / 5 次，再统计稳定性
- 把多次采样做成诊断入口硬依赖

### 原因
这会直接带来：

- 成本上升
- 延迟变长
- 用户等待时间恶化
- 一人维护的调试难度增大

---

# 四、身份链现实裁决

## 4.1 `matched / weak_matched / unresolved` 保留
这套状态保留，不删。

### 原因
它很现实，而且能帮助系统避免伪精确。

---

## 4.2 百度返回名不等于身份真值
正式裁决：

# **百度识别名称只是一条候选输入**
# **身份真值必须经过 Taxonomy 匹配裁定**

---

## 4.3 `taxonomy_match_status` 与 `identity_resolution_status`
虽然文档层已区分，但第一版开发必须注意：

### 现实提醒
这两个字段在很多记录里会高度相似。  
第一版不要围绕这两个字段做过度复杂的业务分支。

### 第一版推荐用途
- 追溯
- 调试
- 日志分析

### 第一版不推荐
- 拿它们做大量状态机跳转条件

---

# 五、视觉入口与接纳层现实裁决

## 5.1 器官槽位提示的地位
你当前的产品结构化槽位设计是有价值的，例如：

- 图1 叶片
- 图2 茎
- 图3 根 / 根颈
- 图4 全株
- 图5 花 / 果 / 其他

### 正式裁决
第一版中，槽位只能视为：

# **强提示**
不是：
# **事实真值**

### 原因
用户会：
- 上传错图
- 放错槽位
- 重复上传
- 把全株图当局部图

所以实现上必须允许：

- 槽位提示与模型识别不一致
- 模型仍可反向给出器官弱提示

---

## 5.2 接纳判定层保留，但对象边界要轻量
当前四级接纳结果保留：

- `formally_admitted`
- `candidate_retained`
- `explanation_retained`
- `direct_reject`

### 第一版裁决
保留这四级，但不要做过于复杂的多对象细粒度接纳系统。

### 第一版建议重点对象
优先对以下对象做清晰接纳：
- 症状候选
- 图像质量结果
- 模式候选（弱）
- 路由建议（弱）

### 第一版不要做
- 对每一种对象都建立一套复杂独立接纳策略
- 接纳规则表过度外部化

---

## 5.3 route hint 的现实执行口径
正式裁决继续保留：

# **route hint 只影响流程优先级**
# **不改变事实层先落位**

### 第一版实现要求
只要条目已被：
- `formally_admitted`

就先进入：
- `observed_evidence_set`

之后再根据：
- `retake_first`
- `ask_first`
- `uncertain_prepare`
- `standard_flow`

决定后续流程优先级。

### 原因
否则 route hint 会反向污染事实层，破坏体系边界。

---

# 六、追问与补图的现实裁决

## 6.1 第一版必须限制交互轮数
这是当前最重要的产品现实裁决之一。

### 正式裁决
第一版严格限制：

- **最多 1 次补图**
- **最多 2 轮追问**

超过后直接进入：
- 保守收敛
- 不确定结论
- 或低风险输出

### 原因
用户不是标注员，也不会无限配合。

---

## 6.2 优先问高收益问题
第一版问题队列不能太长。

### 第一版优先问题
- 是否长期存在
- 是否近期突然出现
- 是否扩散
- 是否可擦除 / 可去除
- 是否伴随环境变化
- 是否需要补一张更清晰图

### 第一版不做
- 过多专业问题
- 过长多分支诊断问卷

---

## 6.3 补图不应成为默认主路径
虽然系统支持补图，但第一版不应让补图成为默认流程。

### 正式裁决
补图只能在这些情况下被优先触发：
- 图像质量明显不足
- 身份链确实无法推进
- 当前继续提问收益明显低于补图收益

### 不允许的倾向
- 一旦不确定就默认让用户补图
- 把补图当作主要解决方案

---

# 七、结论层现实裁决

## 7.1 问题性结论对象：第一版主做
这个是主战场，应优先保证。

---

## 7.2 不确定结论对象：必须保留
第一版一定要保留不确定结论。  
否则系统会被迫把大量证据不足的情况硬判成问题。

---

## 7.3 非问题性结论对象：白名单化
这是关键现实裁决。

### 正式裁决
第一版只允许少数**非常稳**的非问题性结论：

- 正常老叶老化
- 稳定正常斑纹
- 长期稳定正常形态

### 第一版不做
- 大量非问题性分类
- 复杂正常形态库
- 细粒度正常 / 异常交界判断

### 原因
非问题性结论一旦误判，风险很大。  
第一版只做少数白名单，最稳。

---

# 八、属级养护基线与动作建议现实裁决

## 8.1 `genus_care_profiles` 保留并落表
这个方向保留。

### 原因
它对：
- 宿主背景
- explanation
- 行动建议
- 后续长期数据能力

都有价值。

---

## 8.2 第一版动作建议必须软化
这是关键裁决。

### 正式裁决
第一版动作建议只能输出：

# **软建议**
不能输出：
# **处方型硬指令**

### 允许表达
- 更偏向
- 通常建议
- 当前更应优先检查
- 可先尝试从……调整

### 不允许表达
- 必须 7 天浇一次
- 一定是缺水
- 必须立即施肥
- 已确定就是……导致

### 原因
属级基线天然有局限：
- 属内差异
- 品种差异
- 盆土差异
- 季节差异
- 室内环境差异

---

## 8.3 第一版不做 care overrides
虽然文档里允许 future overrides：

- species care overrides
- plant identity care overrides
- horticultural variant overrides

### 正式裁决
第一版不实现这些覆盖层。  
当前只保留概念，不落代码。

### 原因
一人项目第一版没必要引入多层养护覆盖系统。

---

# 九、运行时对象与持久化现实裁决

## 9.1 第一版 SQL 化对象
只坚持当前已经收口的最小集合：

### 必须 SQL 化
- `visual_call_batches`
- `plant_identity_resolution_records`
- `visual_raw_image_records`
- `visual_normalized_image_results`
- `visual_admission_records`

### 建议 SQL 化
- `visual_call_aggregate_results`
- `visual_supervision_records`

---

## 9.2 第一版不强制 SQL 化的对象
继续维持轻量策略：

- `normalized_input`
- `observed_evidence_set`
- `hypothesis_pool`
- `outcome_pool`
- `question_queue`
- `stop_state`
- `diagnostic_trace`

### 正式裁决
第一版这些对象可先放：
- 内存
- cache
- trace
- 会话态结构体

不强制建表。

### 原因
否则第一版会过度工程化。

---

## 9.3 `visual_supervision_records` 只是留痕表
第一版不要把它当：

- 自动训练闭环中心
- 全量质量治理平台
- 自动优化中枢

### 正式裁决
它当前只是：
# **留痕表**
供后续分析使用。

---

# 十、第一版不应做成硬规范的内容

以下内容虽然方向没错，但第一版不应硬做：

## 10.1 模式候选强规则化
第一版不做成强规则触发器。  
最多做 explanation / 排序辅助。

## 10.2 问题回流纠正重状态机
第一版只做必要留痕和覆盖。  
不做精细可回放纠正图谱。

## 10.3 运行时 8 大对象全部精致化
第一版不做。

## 10.4 复杂人工审核流
第一版不做。

## 10.5 高成本模型重复调用
第一版不做。

---

# 十一、给 Codex 的直接实现指令

## 11.1 百度实现原则
- 只做植物名称候选输入
- 不做身份真值裁定

## 11.2 混元实现原则
- 只做候选生成器 + 粗筛器
- 不做最终裁判

## 11.3 交互实现原则
- 最多 1 次补图
- 最多 2 轮追问

## 11.4 结论实现原则
- 问题性：主做
- 不确定：必须保留
- 非问题性：白名单化

## 11.5 养护建议实现原则
- 使用 `genus_care_profiles`
- 但只输出软建议
- 不输出硬处方

## 11.6 工程实现原则
- 先把建表 + 导入链跑通
- 运行时对象轻量化
- 不做复杂治理闭环
- 不做复杂人工审核流
- 不做高成本稳定性采样

---

# 十二、一句话总裁决

**第一版开发必须按“现实可落地版本”执行：模型降级为候选生成器、追问次数严格上限、非问题性结论白名单化、养护建议软化、运行时对象轻量化、禁止高成本稳定性采样。这样系统才是真正可开发、可维护、可迭代的。**
