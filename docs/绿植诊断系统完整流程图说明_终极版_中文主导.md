# 绿植诊断系统完整流程图说明（终极版，中文主导）

## 1. 这次修复了什么

### 修复 1：上游并行链汇合关系
本版明确写死：

- **植物身份主链路**
- **AI视觉症状主链路**

属于：

# **上游并行可独立完成的链路**

含义是：

- 概念上并行
- 实现上可并行、可异步完成
- 谁先完成都可以先暂存
- 最终统一汇入：
  - **上游结果汇合点**
  - 再进入 **输入归一处理**

这样可以避免把两条链误读成“必须严格同步完成”或“必须严格串行执行”。

### 修复 2：停止条件与输出资格拆分
本版明确拆成两个节点：

1. **是否满足停止条件**
2. **停止后是否满足输出资格**

这意味着：

- 停止不等于一定可输出
- 停止后若仍不满足输出资格，系统仍可返回追问 / 补图 / 重算
- 只有停止且满足输出资格，才进入最终结论分类与对外输出

## 2. 如何阅读本图

建议按这个顺序看：

### 第一步：看主干
先看从：
- 用户输入
- 到身份链 / 症状链
- 到接纳判定
- 到输入归一
- 到挂接
- 到问题竞争
- 到追问回流
- 到结论收敛
这一条主干。

### 第二步：看并行上游
注意：
- 植物身份主链路
- AI视觉症状主链路
是并行可独立完成的上游链，最终统一汇入“上游结果汇合点”。

### 第三步：看关键分支
重点看：
- 百度识别后 taxonomy 无法命中
- weak_matched / unresolved
- 图像质量不足
- 正式接纳 / 候选保留 / 仅解释保留 / 直接拒绝
- retake_first / ask_first / uncertain_prepare / standard_flow
- identity / genus / family 挂接
- 问题性 / 非问题性 / 不确定结论

### 第四步：看流程硬边界
最后看：
- 什么不是 outcome
- 什么不是 problem taxonomy
- 属级养护基线的合法位置

## 3. 属级养护基线的位置
本版继续固定：

- 它属于 **Taxonomy 侧主数据**
- 第一阶段通过 **属名 + 科名** 挂接
- 它不进入问题主竞争链
- 它只在：
  - 解释组织
  - 环境偏离判断
  - 动作建议阶段
被消费

## 4. 字段中文解释原则
本图中凡出现关键字段，均配中文解释，例如：

- `visual_call_batch_id`：视觉调用批次ID
- `identity_resolution_status`：身份解析状态
- `visual_raw_image_record`：单图视觉原始记录
- `visual_normalized_image_results`：单图视觉标准化结果
- `visual_admission_records`：视觉接纳判定记录
- `route_primary_action`：路由建议主动作
- `observed_evidence_set`：已观察证据集合
- `question_queue`：追问队列
- `plant_identity_id`：植物身份对象ID
- `genus_care_profiles`：属级养护基线表
