# 多图业务融合与当前阶段自有模型技术路线裁决

**v1 · 完整最终版 · 开发收口候选稿**

> 基于当前 A 类正式基线文件的联动增补裁决，仅在不与既有宪法、硬约束、术语表、运行时模型、数据结构和决策流冲突的前提下生效。

## 文档定位

针对“多图是否已被当前体系支持、当前阶段自有模型如何选型、诊断层多图融合规则如何收口”的一次性技术裁决。

## 当前阶段模型前提

自有模型训练平台明确限定为 **Hugging Face AutoTrain 的 image classification 任务**；样本粒度按**单样本单图**处理。

## 多图实现主线

采用“**单次提交、多图批次、单图识别、聚合结果、证据入集、诊断收敛**”的证据优先路线，不采用黑盒端到端一步诊断。

## 文档效力

本文件是对现有 A 类正式基线的联动增补；在被正式纳入基线清单前，不替代原有 A 类文件。

## 一、文档目的与适用范围

本文件用于把当前业务体系中的多图能力、当前阶段自有模型技术边界、诊断层多图融合规则、工程实现路径与增补代价，一次性收口为可直接供设计、实现、评审与联调使用的统一版本。

本文件不改写既有宪法层文件，不推翻既有术语表，不替代既有运行时模型、核心数据结构、决策流与硬约束；它只在“现有正式基线已经具备多图骨架但尚未完全写死诊断层融合算法”的位置上，补齐缺口，形成可落地的当前阶段技术决策。

## 二、上位约束与不变原则

1. 坚持“**证据优先、解释驱动、结论收敛**”的主轴，不允许任何多图逻辑绕过证据层直接写入诊断排序或诊断结论。  
2. 坚持“**只增不减**”的联动原则；本文件仅补充规则、字段、流程与实现边界，不删除既有正式对象。  
3. 坚持**中文主名优先**；英文只作为字段名、辅助对象名或技术实现映射存在，不得压过中文业务概念。  
4. 坚持**多图不是 outcome、多图不是 diagnosis、多图不是 problem taxonomy**；多图只是输入组织方式与证据获取方式。  
5. 坚持当前阶段自有模型必须服从现实技术边界，不以未来可能采用的 VLM / 自定义训练脚本，反向污染当前版本的工程决策。

## 三、当前阶段总裁决

| 裁决编号 | 问题 | 结论 |
|---|---|---|
| R-01 | 当前业务是否支持多图 | 支持，但当前支持主要体现为“多图输入、多图批次、单图标准化、聚合结果、证据入集、补图与降级路径”；尚缺最终冻结的诊断层量化融合规则。 |
| R-02 | 当前阶段自有模型能否原生单样本多图 | 不能。当前阶段自有模型前提锁定为 Hugging Face AutoTrain 的 image classification 任务，该任务按 image_column / target_column 的单图样本范式组织。 |

| R-04 | 当前阶段推荐路线 | 采用“HF AutoTrain 单图训练 + 多图批次上传 + 服务端并行单图识别 + 业务层融合 + evidence 驱动 diagnosis”的路线。 |

## 四、当前业务对多图融合的支持程度裁定

从现有正式基线看，多图并不是后加的旁路能力，而是已经进入 AI 视觉入口层、运行时模型与诊断主链的正式对象体系。当前体系已经明确承认：同一轮次内一次用户提交多张图会形成一个 `visual_call_batch`；该批次下可以挂接多条单图视觉原始记录、多条单图视觉标准化结果、一个视觉调用聚合结果，以及一组视觉接纳判定记录。

同时，产品侧允许按**叶、茎、根 / 根颈、全株、花 / 果**等槽位上传图片，并将槽位语义定义为正式的**结构化输入提示**。这意味着系统并非只把多图视作文件集合，而是已经将多图上升为具备**器官语义、批次语义、聚合语义和补图语义**的运行时入口。

因此，本文件的基本判断不是“从零开始做多图”，而是“对已经存在的多图骨架做诊断层收口”。现阶段不足不在于能不能上传多张图，也不在于能不能存多张图，而在于诊断排序层尚未把**去重、独立性、跨器官差异加权、跨批次覆盖和聚合输出 contract**完全冻结。

## 五、当前阶段自有模型技术选型裁决

当前阶段自有模型的正式技术选型，明确限定为：**Hugging Face AutoTrain 的 image classification 训练范式**。该裁决不是泛泛描述，而是当前版本的真实落地前提。任何需要原生单样本多图、原生多图问答、原生多图推理的方案，都不应被写入当前阶段的自有模型实现假设。

原因有三：

1. 当前 AutoTrain 的 image classification 任务以单个 `image` 列和单个 `target` 列为基本组织方式；它解决的是“单样本单图分类或回归”，而不是“单样本多图病例级联合分类”。  
2. 当前能在 AutoTrain 中配置的 `batch_size`、`learning rate`、`optimizer`、`scheduler`、`fp16`、`gradient_accumulation`、`early_stopping` 等参数，都只影响训练过程，而不改变单个样本如何被组织。  
3. 现有业务规范本来就以**证据优先和运行时分层**为主轴，更适合把多图视作多来源证据输入，再在业务层收口，而不是提前走黑盒端到端。  

因此，当前阶段的自有模型不应写成“多图模型”，而应写成“**单图症状识别模型**”。多图能力来自业务层，不来自当前训练平台。

## 六、AutoTrain 训练参数的法律地位与边界

当前 AutoTrain 参数集的法律地位，应被明确写成：**训练过程调优参数，而非样本结构定义参数**。

| 参数类别 | 能改变什么 | 不能改变什么 |
|---|---|---|
| `batch_size` / `gradient_accumulation` | 单步吞吐、显存占用、有效批大小 | 不把 1 个样本变成多图病例 |
| `epochs` / `early_stopping` | 训练轮次与停止时机 | 不改变输入样本粒度 |
| `lr` / `warmup_ratio` / `scheduler` / `optimizer` | 收敛速度、稳定性、更新路径 | 不增加多图联合推理能力 |
| `mixed_precision` / `fp16` | 计算效率与显存效率 | 不改变任务范式 |
| `weight_decay` / `max_grad_norm` / `seed` | 正则化、梯度稳定性、可复现性 | 不把 image classification 变成 VLM 训练 |

所以，任何试图用参数层设置来突破单样本单图限制的写法，都应在当前版本中被明确禁止。当前阶段若需要多图原生样本，必须切换到另一类训练框架；但那属于后续阶段，不属于本裁决生效范围。

## 七、当前阶段的正式技术路线

当前阶段的正式路线，应写成“**单次提交、多图批次、单图识别、聚合收口、证据入集、诊断更新**”。产品层看起来是一轮多图诊断，模型层实际上是多张单图的并行识别，运行时层则把这些结果组织成证据并驱动 diagnosis。

### 必须满足的流程约束

1. 用户可在同一轮次内一次提交多张图，且允许按器官槽位上传。  
2. 系统必须生成新的 `visual_call_batch`，并为每张图生成 `visual_raw_image_records` 与 `visual_normalized_image_results`。  
3. 每张图的识别结果必须保留**器官绑定、图像质量、route hint、症状候选与置信修正来源**。  
4. 系统必须生成 `visual_call_aggregate_result`，对同批次多图做**覆盖度、重复度、质量、器官一致性与聚合症状判断**。  
5. 只有通过接纳规则的对象才能进入 `observed_evidence_set`；候选保留对象不得越权写成正式 evidence。  
6. `diagnosis` 只能消费 `evidence` 与 `derived evidence`，不得直接消费模型高分裸结果。

## 八、诊断层多图融合规则（本文件新增收口）

以下规则为本文件最核心的新增内容。它们的目标不是再造一套新系统，而是把现有多图骨架中最容易引发实现漂移的地方写死。

### 8.1 同批次重复视角抑制规则

同一视觉调用批次内，如果多张图高度可能描述的是**同一器官、同一局部、同一症状**的重复视角，则这些结果不得线性叠加。系统必须先进行“**重复视角归并**”，再决定是否对该症状进行增强。重复视角归并后的默认处理是：**提高该症状的稳定性判断，但不按图片数量直接倍增其对 diagnosis 的支持强度**。

重复视角的典型条件：

- 器官一致  
- 空间区域高度相近  
- 症状键相同或极相近  
- 拍摄时间相近且主体明显相同  

重复视角组内必须保留**最高质量图**作为主引用图，其余图保留为辅助引用，不得各自独立重复加分。

如果只是在同一器官不同区域多次出现同类症状，可判定为“**同器官多点支持**”；此时允许有限增强，但仍不得按张数线性累计。

### 8.2 跨器官独立支持规则

不同器官的同向证据，原则上比同器官重复视角更具独立性，但也必须受规则约束。对于器官敏感型问题，系统应优先承认与该问题主作用器官更相关的证据；其他器官证据只作为辅助支持，不得反客为主。

| 问题判断方向 | 高优先器官 | 低优先器官 | 当前阶段规则 |
|---|---|---|---|
| 根腐 / 烂根方向 | 根 / 根颈、茎基部 | 叶片、全株 | 叶片黄化只能辅助；根与根颈证据优先级更高 |
| 日灼 / 环境灼伤方向 | 叶片、全株 | 根 | 全株分布与叶片局部征象可互相补强 |
| 虫害方向 | 叶片背面、嫩梢、茎部 | 根 | 局部虫咬或虫体证据优先于泛黄等弱间接现象 |
| 花果异常方向 | 花 / 果、相关连接部位 | 单纯叶片 | 花果图缺失时不得仅凭叶图强推花果问题 |

上表不是为了把问题权重参数写成死数字，而是为了先冻结**主从关系**：跨器官支持可以增强稳定性，但器官优先级不能被打乱。当前阶段如果尚未建立完整权重矩阵，至少必须先实现“**主器官优先、辅助器官有限支持**”的保守策略。

### 8.3 跨批次覆盖与继承规则

补图后必须形成**新的视觉调用批次**，不得回写旧批次。新的批次可以覆盖旧批次，但覆盖必须是**显式、可追溯、局部可解释**的。

- 当新批次仅补某一器官时，默认只优先覆盖该器官相关的低质量或弱稳定证据，不得无条件抹除其他器官的有效证据。  
- 当新批次与旧批次冲突时，优先级按“**人工确认 > 更高质量且更针对性的器官证据 > 更晚批次的同等级证据 > 旧主结果稳定保持**”的顺序处理。  
- 任何覆盖都必须记录 `superseded target`、覆盖原因、覆盖时间和覆盖来源批次。  

### 8.4 质量降级与补图优先规则

多图系统中的正确动作，不是图多就硬判，而是图多但质量不足时主动降级。当前阶段必须把“**质量不足时先补图、再诊断**”写成硬行为。

- 当聚合 `analyzability` 为 `marginal` 或更低时，应优先生成补图问题，而不是继续细分 diagnosis 提问。  
- 当器官提示存在但图像质量不够时，器官槽位语义只能降低 follow-up 成本，不能越权代替图像事实。  
- 当关键器官缺失时，允许形成保守 explanation 或不确定 outcome，但不得强推高风险问题。  

### 8.5 接纳与候选保留边界

多图聚合后，不是所有聚合症状都自动进入 evidence。当前阶段必须继续沿用“`formally admitted`”与“`candidate retained`”的分层。

- 多图重复出现、器官清晰、质量合格、映射稳定的症状，可 `formally admitted`。  
- 仅单图弱出现、长尾噪声明显、器官未明或质量不足的症状，只能 `candidate retained`。  
- `route hint` 只进入流程层和 `question_queue`，不得反写为事实层或 diagnosis evidence。

## 十、前端与后端的正式实现方式

当前阶段不要把“多轮单图识别”实现成用户可见的多次串行等待。对用户而言，应尽量表现为**一次批次提交**；对服务端而言，才是**单图模型并行识别后再聚合**。

### 实现要求

- **前端**：一次性上传多张图，并携带器官槽位语义、拍摄顺序、是否补图等结构化信息。  
- **网关 / 云函数**：以一次请求创建一个 `visual_call_batch`，统一鉴权、统一存储、统一追踪。  
- **模型服务**：按单图拆分任务并行执行，不采用前端逐张串行调用。  
- **聚合层**：等待同一批次的单图结果收齐或达到最小可用阈值后，生成 `visual_call_aggregate_result`。  
- **运行时层**：统一做接纳判定、evidence 写入、`question_queue` 更新与 diagnosis 重算。  

该实现方式的本质是：**底层是多张单图，外层是一次多图业务**。这样既尊重当前 AutoTrain 的技术边界，又尽量避免用户体验被“多轮”这个词拖慢。

## 十一、性能判断与用户体验裁决

多轮单图相较于未来理想化的单轮多图原生模型，理论上会多出若干网络往返和多次前向计算，因此在纯理论层面通常更慢。但在当前体系里，真正决定体感速度的关键不是“是不是单图模型”，而是“有没有把服务端实现成串行”。

如果把 4 张图做成前端 4 次顺序请求，速度一定明显变差；但如果做成**一次批次上传、服务端并行识别、一次聚合返回**，那么用户体感延迟会被控制在“**最慢那一张图的处理时间 + 聚合时间 + 一次批次开销**”的范围内，通常不会与单轮多图拉开数量级差距。

因此，当前阶段的正式性能结论应写成：

> 多图业务允许采用单图模型，但必须配套“批次上传 + 服务端并行 + 一次聚合返回”的工程策略。任何前端串行逐张识别的实现，都应视为不符合本文件要求。

## 十二、扩展代价评估

| 扩展项 | 代价判断 | 原因 | 优先级 |
|---|---|---|---|
| 补齐诊断层多图融合规则 | 中 | 主要是规则和字段收口，不是架构翻新 | 最高 |
| 补齐聚合对象 contract | 中低 | 已有 `visual_call_aggregate_result`，只需增补字段与写法 | 高 |
| 前端器官槽位与补图引导 | 中 | 有明显产品收益，且可降低 follow-up 成本 | 高 |
| 切换到原生多图 VLM 训练 | 高 | 需要切换训练框架、数据组织方式与评估体系 | 低（当前阶段不做） |

综合判断：当前版本从“已经有多图骨架”走到“可上线的多图业务融合”，代价属于**中等**，而不是重构级。最贵的部分——器官槽位、视觉调用批次、单图标准化、运行时 evidence 主轴——都已经存在。当前真正要补的是**规则冻结、字段补齐和工程实现收口**。

## 十三、当前阶段明确不做的事情

1. 不把当前 AutoTrain 单图模型包装成“原生多图模型”。  
2. 不在当前版本承诺端到端黑盒多图 diagnosis。  
3. 不允许模型高分字段直接裸写 `observed_evidence_set`。  
4. 不允许把器官槽位语义、`route hint`、聚合质量等流程层对象越权写成 outcome 或 problem taxonomy 对象。  
5. 不允许在缺少关键器官图或关键证据冲突未解时，过早给出强动作建议。  

---

## 十四、最终绑定决策

为确保实现端、评审端和后续 Codex / Claude / 人工开发在一个版本上工作，本文件最后把当前阶段的绑定决策一次性写死如下。

1. **绑定决策 1**：当前阶段自有模型训练平台，正式限定为 Hugging Face AutoTrain 的 image classification 任务。  
2. **绑定决策 2**：当前阶段样本粒度，正式限定为单样本单图；多图能力来自业务层，不来自训练层原生多图。  
3. **绑定决策 3**：当前阶段多图主线，正式限定为单次提交、多图批次、服务端并行单图识别、聚合结果、证据入集、诊断更新。  
4. **绑定决策 4**：多图融合必须遵守证据独立性、重复视角抑制、跨器官主从关系、跨批次显式覆盖、质量不足优先补图。  
5. **绑定决策 5**：未来若要原生单样本多图训练，必须另立技术路线文档，不得把未来路线偷渡进当前阶段实现。
