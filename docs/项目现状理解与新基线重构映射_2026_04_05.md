## 3. 一句话结论

当前项目已经有一条可运行的旧链路，但它的核心中枢仍然是：

-   `plant_catalog + plant_aliases + genus_care_profile`
-   `identify_sessions`
-   `diagnosis_sessions + diagnosis_symptom_observations + diagnosis_problem_rankings + diagnosis_follow_ups + diagnosis_result_snapshots`

而新正式基线要求的核心中枢已经变成：

-   Taxonomy 静态层
-   植物身份主表 / 命名归一 / 身份解析层
-   AI 视觉入口批次与规范化层
-   `observed_evidence_set -> route_primary_action -> diagnosis/outcome`

所以后续开发不是补丁式修改，而是一次有边界的“核心模型迁移”。

## 4. 当前项目的真实结构

### 4.1 前端层

当前前端是 UniApp 微信小程序壳，应用在 `src/App.vue` 中直接执行 `wx.cloud.init`，说明运行形态仍是小程序 CloudBase 前端直连模式。

用户主入口不是单独的“诊断页”，而是首页植物列表页：

-   首页：`src/pages/index/index.vue`
-   诊断弹窗：`src/components/DiagnosePopup.vue`
-   植物详情页：`src/pages/plant-detail/plant-detail.vue`
-   添加植物页：`src/pages/add-plant/add-plant.vue`

这里最关键的结构判断是：

1.  首页持有用户植物列表，并直接从植物卡片上触发诊断。
2.  真正的诊断交互发生在 `DiagnosePopup` 中。
3.  `src/pages/diagnose/diagnose.vue` 更像结果承接页或旧残留页面，不是现在的主要链路入口。

### 4.2 现有植物主数据链

当前“植物是谁”这件事，不是通过新基线里的 identity 体系实现，而是围绕以下旧模型展开：

-   `plant_catalog`
-   `plant_aliases`
-   `genus_care_profile`

核心访问层在：

-   `cloudfunctions/layer/utils/plant-knowledge.js`

这个文件承担了当前系统最重要的旧知识库职责：

1.  列目录植物。
2.  按别名/模糊名匹配植物。
3.  从目录信息与属级养护表拼出前端所需的植物展示与养护字段。
4.  创建 `user_plant_instances`。

也就是说，当前代码里的“taxonomy/identity/care”并没有真正分层，而是被压平到了 `plant_catalog` 及其周边 SQL 查询里。

### 4.3 添加植物链

当前“添加植物”的主链是：

1.  在 `src/pages/add-plant/add-plant.vue` 中搜索常见植物，或发起 AI 识别。
2.  AI 识别调用 `identify-http`。
3.  `identify-http` 调百度识别接口。
4.  百度返回名称后，服务端通过 `findCanonicalPlantMatch` 在旧 `plant_catalog` 体系里查匹配。
5.  最终创建 `user_plant_instances`。

这条链的本质是：

-   图像识别结果直接被尝试映射为旧 `plant_catalog` 的 `plant_id`
-   识别过程只留下 `identify_sessions`
-   没有新基线要求的视觉批次、原始返回、规范化对象、准入记录、身份解析记录

### 4.4 诊断链

当前诊断主链由 `cloudfunctions/diagnose-http/app.js` 驱动，结构上是标准的旧三段式：

1.  `start`
2.  `answer`
3.  `result / history / feedback`

其内部逻辑特征非常明确：

1.  首轮支持图片输入。
2.  首轮在没有显式 `observedSymptoms` 时，会走 LLM 图像诊断抽取症状。
3.  进入 follow-up 后，不允许补图。
4.  服务端会把症状、问题排序、追问题、最终结果分别落进旧的多张诊断表。

当前诊断链仍然明显带着“旧诊断引擎向新接口外观兼容”的痕迹，例如：

-   `adaptObservedSymptoms`
-   `adaptLegacyFollowUpAnswers`
-   `buildLegacyHttpSuccess`
-   `normalizeHistoryDetail`

这说明目前代码层已经在做“接口壳升级”，但核心事实模型还没有切到新正式基线。

### 4.5 前端诊断结果契约

前端当前消费的诊断对象仍围绕以下结构组织：

-   `observedSymptoms`
-   `rankings`
-   `followUps`
-   `finalResult`

关键处理在：

-   `src/http-functions/diagnose/client.js`
-   `src/utils/diagnose-flow.js`

也就是说，前端虽然已经有“分轮问诊”和“阶段化结果”的外观，但它理解的仍然是旧问题模型，不是新文档要求的 evidence-first 与 route-first 数据契约。

### 4.6 数据工具链

仓库中已经存在一套真实可执行的数据工具链：

-   `src/cli.js`
-   `src/data-system/importer/excel-importer.js`
-   `src/data-system/diff/diff-engine.js`
-   `src/data-system/publish/publish-engine.js`
-   `src/data-system/validator/schema-validator.js`

这非常重要，因为它意味着后续开发不是从零做“数据发布系统”，而是：

-   复用现有 CLI / import / diff / publish / rollback 骨架
-   把其配置对象、目标表、字段映射、校验逻辑迁到新正式 SQL 方案

## 6. 旧实现与新正式基线的差距

### 6.1 植物身份中心不同

旧实现的植物身份中心是 `plant_catalog`。

新正式基线要求的身份中心则是：

1.  Taxonomy 静态定义层
2.  植物身份主表
3.  命名归一层
4.  身份解析记录层

这意味着后续不能再把“目录植物行”直接当作最终身份真相。

### 6.2 AI 视觉入口不同

旧实现的视觉入口只有一条简化链：

1.  图片
2.  百度识别
3.  名称匹配
4.  写 `identify_sessions`

新正式基线要求的视觉入口是一整层对象：

1.  调用批次
2.  原始返回
3.  规范化结果
4.  准入结论
5.  身份解析
6.  与诊断 / 用户植物实例的挂接

所以 `identify-http` 不会只是“换模型”这么简单，而是职责重写。

### 6.3 诊断事实层不同

旧实现是“症状 -> 问题排序 -> 追问 -> 最终结果”的旧诊断中心。

新正式基线要求的是：

1.  先形成 `observed_evidence_set`
2.  再形成 `route_primary_action`
3.  再进入 diagnosis / outcome 层

这意味着旧的 `diagnosis_sessions`、`diagnosis_problem_rankings`、`diagnosis_follow_ups` 这一套结构，不能继续充当新系统的核心事实层。

### 6.4 前端协议不同

当前前端拿到的是“旧诊断外观对象”，例如：

-   `followUps`
-   `rankings`
-   `observedSymptoms`
-   `finalResult`

新正式基线下，前端至少要适配：

1.  身份解析状态与植物身份候选
2.  证据集与准入信息
3.  路由动作
4.  诊断结果或转向动作

也就是说，UI 壳未必要先推倒，但请求与响应契约一定会改。

### 6.5 数据导入目标不同

现有 CLI 能跑，但它针对的是旧表配置。

新正式基线下，Excel 导入、字段归一、发布和回滚都必须面向新 SQL 方案重建映射，尤其是：

-   taxonomy 静态表
-   植物身份主表
-   名称归一表
-   diagnosis / linkage 相关正式表
-   视觉入口相关持久化表

## 8. 面向开发的落地理解

### 8.1 第一阶段目标

第一阶段不是“把所有页面都改漂亮”，而是先建立新的运行骨架：

1.  新 SQL 表结构落地。
2.  新 taxonomy / identity 查询链落地。
3.  新 AI 视觉入口落地。
4.  新诊断会话主链落地。
5.  前端最小可跑通新协议。

### 8.2 第一阶段最小闭环

最小闭环应该是：

1.  用户上传图片。
2.  系统先完成视觉批次记录与规范化。
3.  系统完成植物身份解析。
4.  系统形成 `observed_evidence_set`。
5.  系统根据 `route_primary_action` 进入标准诊断或其他动作。
6.  结果能正确返回前端并完成持久化。

### 8.3 不建议的做法

以下做法会导致后续返工，应避免：

1.  在旧 `plant_catalog` 结构上继续追加“新身份字段”。
2.  在 `identify_sessions` 上继续叠新视觉入口语义。
3.  在旧 `diagnosis_sessions` 上勉强塞入新 route / evidence 语义。
4.  先大量改页面，再回头改数据契约。

## 9. 建议的开发顺序

### 9.1 推荐顺序

1.  以 A 类正式基线文档为唯一依据，冻结新 SQL 实体与字段。
2.  先建设新数据访问层，不在旧 `plant-knowledge.js` 上继续叠加。
3.  重写 `identify-http`，让视觉入口先符合新基线。
4.  重写 `diagnose-http` 的核心会话与持久化逻辑。
5.  再适配前端请求/响应对象。
6.  最后再处理页面级体验与展示优化。

### 9.2 工程拆分建议

实际执行时可以拆成四个并行但边界清晰的子任务：

1.  SQL 与 repository 层。
2.  植物 identity / taxonomy 服务层。
3.  视觉入口与诊断服务层。
4.  前端协议适配层。
