# 主动瘦身计划：以养护类问题为主轴

本文档包依据以下材料整理：

- 当前代码压缩包：`Archive 2.zip`
- 项目规则压缩包：`rules.zip`
- 诊断运行时粗文档：`diagnosis-runtime-code-logic.md`
- 本次会话中关于“主动瘦身、养护类主轴、outcome 路径规划、gate 守卫、LLM prompt 职责边界”的讨论结论

权威优先级：**当前代码 > 项目 rules > 已有运行时说明 > 本次设计讨论**。如果后续实施时发现文档和代码冲突，必须以代码为准，并同步修正文档。


## 一、瘦身总判断

当前系统如果继续沿着“病虫害细分 + 具体问题 ranking”的方向扩张，会越来越难维护。

更适合当前 MVP 的方向是：

```text
主攻养护类问题
输出问题簇或处理方向
减少具体病虫害细分
问诊围绕关键养护事实
结果以用户该怎么处理为中心
```

这不会让复杂度断崖式归零，但会带来结构性降复杂度。

预计效果：

```text
总复杂度下降约 40%～60%
```

下降的主要来源不是删除主链，而是减少覆盖范围、减少路径分叉、减少具体问题竞争、减少结果解释分支。

## 二、为什么主攻养护类问题

真实家庭绿植场景中，用户最常关心的是：

```text
黄叶
萎蔫
掉叶
焦边
不长新叶
徒长
长得慢
叶片发软
盆土久湿
是不是浇多了
是不是缺水
是不是光照不对
还能不能救
```

用户通常不是先关心：

```text
这是哪一种真菌？
是不是细菌叶斑？
具体是哪种虫？
病原学上叫什么？
```

所以 MVP 的用户价值不应是“病名识别”，而应是：

```text
判断植物为什么长不好，并给出安全、可执行的养护调整方案。
```

## 三、瘦身范围

### 1. 保留的主轴

第一阶段主轴：

| 主轴 | 典型问题 | 对应 outcome |
|---|---|---|
| 水分 | 浇多、缺水、盆土久湿、盆土干透 | 积水/根系压力、缺水压力 |
| 光照 | 弱光、徒长、晒伤、换位置后异常 | 光照不足/生长偏弱、晒伤/强光刺激 |
| 生长状态 | 不长新叶、长得慢、枝条细弱 | 生长偏弱/徒长 |
| 叶片状态 | 黄叶、焦边、卷叶、萎蔫 | 黄叶分流、叶尖焦枯/环境压力 |
| 环境 | 通风差、空气干、温度变化 | 环境压力类 outcome |
| 非问题 | 老叶代谢、艺斑、新叶嫩色 | 自然代谢/正常斑纹 |

### 2. 保留为分支的次轴

| 次轴 | 处理方式 |
|---|---|
| 叶斑类 | 作为问题簇，不细分真菌/细菌/炭疽。 |
| 疑似虫害 | 作为问题簇，先看虫体/虫网/黏液/壳状物。 |
| 结构损伤 | 作为非病害问题簇，避免误喷药。 |
| 范围外异常 | 作为不确定或池外观察，不强行诊断。 |

### 3. 暂缓的内容

第一阶段暂缓：

```text
具体真菌病名
具体细菌病名
大量具体虫害分类
复杂宿主适配矩阵
细粒度营养元素缺乏诊断
复杂多轮问诊树
病害百科式结果页
```

## 四、推荐 MVP 植物侧重点

为了配合养护类主轴，第一版植物库建议优先覆盖高频室内植物/品类：

```text
1. 绿萝
2. 发财树
3. 金钱树
4. 富贵竹/开运竹/观音竹
5. 多肉植物
6. 白掌
7. 红掌
8. 虎皮兰/虎尾兰
9. 龟背竹/春羽/蔓绿绒类
10. 竹芋/粗肋草/彩叶芋类
```

这些植物共同高频问题是水、光、湿度、通风、黄叶、萎蔫、焦边、生长弱。它们比具体病虫害更适合 MVP 路径规划。

## 五、问题库瘦身

### 1. 具体问题减少

建议第一阶段只保留：

```text
8～12 个可输出 outcome，运行时通过多候选集合逐步收窄
```

不要保留大量具体 problem 参与最终竞争。

### 2. 具体病虫害降级

降级规则：

| 原本具体问题 | MVP 处理 |
|---|---|
| 真菌叶斑、细菌叶斑、炭疽病 | 合并为叶斑类问题。 |
| 红蜘蛛、介壳虫、蓟马、蚜虫 | 合并为疑似虫害痕迹。 |
| 真菌蚊 | 不作为主 outcome，可作为积水/土表潮湿路径下的伴随观察方向；只有土表小飞虫等正式证据通过展示 gate 后才可前端可见。 |
| 各种营养元素缺乏 | 暂时合并为生长偏弱/养护性黄叶待分流。 |

### 3. 非问题要保留

不要砍掉非问题结果。

原因：很多用户拍的并不是病害，而是：

```text
自然老叶
新叶嫩色
正常斑纹
艺斑
机械旧伤
环境适应期少量掉叶
```

如果没有非问题，系统会被迫把正常现象说成问题。

## 六、问诊瘦身

### 1. 每个 outcome 只保留 1～2 个关键问题

示例：

| outcome | 关键问题 |
|---|---|
| 积水/根系压力 | 盆土是否长期潮湿？茎基部/根是否发软、有异味？ |
| 缺水压力 | 盆土是否明显干透？叶片发软时土是干的还是湿的？ |
| 光照不足 | 是否长期在弱光处？新枝是否细长、叶间距变大？ |
| 晒伤/强光刺激 | 最近是否突然暴晒或换到强光处？异常是否集中在受光面？ |
| 叶尖焦枯/环境压力 | 是否靠近空调/暖气、空气很干？最近是否施肥偏多？ |
| 叶斑类问题 | 斑点是否扩散？是否经常喷水或通风差？ |
| 疑似虫害痕迹 | 叶背/茎节是否看到虫、网、黏液、壳状物？ |
| 自然代谢/非问题 | 是否只有底部少量老叶发黄，新叶正常？ |

### 2. 最多 2 轮问诊

规则：

```text
最多追问 2 轮。
每轮 1 个问题；问题优先服务于区分当前候选 outcome，而不是服务某一个最高排名 outcome。
仍不能闭合，则输出不确定 + 保守建议。
```

### 3. 问题不再追求动态最优

当前系统有多来源候选题和收益停止策略。瘦身后优先：

```text
当前 active route 的 gate 缺什么，就问什么。
```

旧问题排序作为 fallback。

## 七、结果页瘦身

结果页不做复杂病害报告，而是固定四层：

```text
1. 更像什么处理方向
2. 为什么这么判断
3. 今天怎么做
4. 接下来 3～7 天怎么观察，什么不要做
```

示例：

```text
判断：更像积水/根系压力。
原因：叶片发黄，同时你确认盆土长期潮湿。
今天：先暂停浇水，放到通风散射光处，检查盆底是否积水。
3 天内：观察土壤是否逐步变干，叶片是否继续发软。
避免：不要继续浇水，不要马上重肥，不要闷在不通风处。
```

## 八、行动建议体系优先

瘦身后最重要的不是诊断名，而是建议是否安全。

行动建议应绑定：

```text
outcome + 风险等级 + 植物养护基线 + 用户回答证据
```

不要只绑定具体 problem。

### 建议模板

每个 `actionProfile` 至少包含：

```text
今天要做什么
3 天内观察什么
7 天内复查什么
不要做什么
何时需要重新拍照或人工判断
```

## 九、不要砍的机制

以下机制必须保留：

| 机制 | 为什么不能砍 |
|---|---|
| 正式证据准入 | 防止视觉模型原始输出越级。 |
| 不确定结果 | 防止系统被迫胡说。 |
| 用户否定路径约束 | 防止越问越偏。 |
| 黄叶分流 | 黄叶对应的处理动作可能相反。 |
| 孔洞分流 | 孔洞可能是虫害、旧伤、机械损伤。 |
| 宽泛斑块守卫 | 宽泛斑块不能直出具体病名。 |
| 非问题结果 | 防止把正常现象说成病害。 |
| 行动建议冲突检查 | 防止同一 outcome 内同时给补水和停水。 |

## 十、复杂度变化评估

| 模块 | 当前复杂度 | 瘦身后变化 |
|---|---:|---|
| 具体问题库 | 高 | 大幅下降。 |
| 问题 ranking | 高 | 明显下降，变成候选排序。 |
| 问诊生成 | 高 | 明显下降，改为 route gate 问题。 |
| 结果解释 | 高 | 大幅下降，围绕处理方向。 |
| 数据建模 | 中高 | 短期上升，因为要补 route/gate/action profile。 |
| 守卫系统 | 高 | 下降但不会消失。 |
| 测试审计 | 高 | 明显下降，可按 outcome route 覆盖。 |
| 前端交互 | 中高 | 下降，结果结构更固定。 |

最终判断：

```text
不是断崖式下降到极简系统，而是从复杂诊断引擎降为窄而稳的养护路径引擎。
```

## 十一、黄金样例覆盖建议

第一阶段建立 80～120 个黄金样例：

| 场景 | 建议数量 |
|---|---:|
| 黄叶 + 积水 | 10 |
| 黄叶 + 缺水 | 10 |
| 黄叶 + 自然老叶 | 8 |
| 萎蔫 + 土湿 | 8 |
| 萎蔫 + 土干 | 8 |
| 光照不足/徒长 | 8 |
| 晒伤/强光 | 8 |
| 叶尖焦枯 | 8 |
| 叶斑类 | 10 |
| 疑似虫害 | 8 |
| 结构损伤/旧伤 | 8 |
| 正常斑纹/非问题 | 8 |
| 不确定/冲突 | 10 |

每个样例验收：

```text
是否进入正确 route
是否问了关键问题
是否正确闭合或不确定
行动建议是否不冲突
是否没有泄漏内部 ranking
```
