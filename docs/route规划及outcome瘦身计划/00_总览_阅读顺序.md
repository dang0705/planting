## 一、文档包目标

这组文档要解决三件事：

1. 把当前诊断流从“问题排名驱动”改造成“多候选 outcome 收敛式路径规划驱动”的实施计划落地。
2. 把本次会话中形成的“outcome、route、gate、问题簇、行动建议中心化”的设计写成长期知识文档。
3. 把主动瘦身方向明确成可执行范围，避免项目继续沿着“具体病虫害百科式诊断”扩张。

这里的核心判断是：

> 当前系统不应该继续追求大量具体问题排名，也不应该在第一轮就锁定某个唯一 outcome。系统应先形成多个候选 outcome，再通过 route group、route、gate 和问诊回答逐步增强、削弱、阻断与收窄，最终给前端展示 1～3 个用户能理解的处理方向。问诊不是为了把分数堆到某个最高问题，而是为了区分当前候选 outcome，尤其是区分“补水/停水”“增光/避光”“处理/不处理”这类行动建议可能冲突的方向。

## 三、一句话总方案

将当前的：

```text
视觉证据 + 问诊证据 + 先验 → 多个 problem ranking → 最高问题通过守卫 → 输出
```

改成：

```text
正式证据 → 候选 outcome 集合 → route group 分流 → route/gate 增强、削弱、阻断 → 问关键分流题 / 收窄到 1～3 个可见 outcome / 不确定 → 输出处理方向
```

其中：

- `outcome` 是用户可理解的处理方向或问题簇；它可以是主方向，也可以是伴随观察方向。
- `candidateOutcomeSet` 是运行时维护的候选 outcome 集合，内部可多于前端展示数量。
- `routeGroup` 是同一入口症状下的分流路径组，例如黄叶可同时打开积水、缺水、光照不足、自然代谢等候选。
- `route` 是某个候选 outcome 的可达证据路径。
- `gate` 是 route 是否增强、削弱、阻断、需要继续问或可展示的判断条件。
- `ranking` 在过渡期只作为候选排序和审计依据，不再直接决定最终答案。

## 四、最重要的保留项

主动瘦身不等于砍掉安全骨架。以下机制不能删除：

1. 正式观察证据准入机制。
2. 视觉原始结果不得直接输出最终诊断。
3. 用户否定后的路径约束。
4. 合法“不确定”结果。
5. 不确定结果不得泄漏内部最高问题。
6. 黄叶、孔洞、宽泛斑块、宽泛生长异常等高频误判场景的分流 gate。
7. 行动建议不得互相冲突。

## 五、最终架构目标

MVP 目标不是“识别一百种病虫害”，而是：

```text
用户拍照后，系统能围绕少数高频养护问题，快速判断更像哪类处理方向，并给出安全、可执行、不会互相打架的建议。
```

建议第一阶段控制为：

- 8～12 个可输出 outcome，运行时内部 active outcome 通常控制在 5～7 个以内。
- 前端问诊中和最终页最多展示 1～3 个 outcome；其中 1 个可以是主方向，0～2 个可以是伴随观察方向。
- 每个入口场景配置 1 个 route group；每个 route group 下通常保留 2～5 个候选 outcome 分支。
- 每条 route 最多 1～2 个关键 gate。
- 最多 2 轮问诊，每轮 1 个问题。
- 输出以“今天做什么、3 天内观察什么、7 天内复查什么、不要做什么”为中心。
