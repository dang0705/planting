## 一、数据层改造目标

route 模式要求数据层能回答这些问题：

```text
有哪些可输出 outcome、候选 outcome 和伴随观察方向？
哪些入口场景会打开 route group？
每个 route group 下有哪些候选 outcome？
每个 outcome 有哪些可达 route？
每条 route 需要哪些 gate？
每个 gate 需要哪些证据或回答？
回答 yes/no/unknown 后支持、削弱、排除还是转向？
收窄到 1～3 个前端可见 outcome 后给用户什么行动建议？
多个可见 outcome 的行动建议是否相容？
```

当前数据层主要围绕：

```text
problem
symptom
symptom_problem_evidence
question
question_option_mapping
question_strategy
diagnosis_result_explanations
```

route 模式下，不建议一开始推翻这些表，而应做兼容扩展。

## 三、Phase 1：复用 `problems` 作为 outcome 表

### 为什么先复用

当前主链大量依赖 `problem_key`：

```text
candidateProblemKeys
rankings.problemKey
question_strategy_v5_real.problem_key
diagnosis_result_explanations.problem_key
symptom_problem_evidence.problem_key
```

如果立即全局改成 `outcome_key`，风险很高。

第一阶段推荐：

```text
数据库字段仍叫 problem_key
业务定义改成 outcome_key
```

### `problems` 中建议新增/明确字段

如果允许改表，可新增：

| 字段 | 中文含义 | 示例 |
|---|---|---|
| `outcome_type` | outcome 类型 | `care_cluster`、`problem_cluster`、`non_problematic`、`uncertain` |
| `outcome_category` | 所属大类 | `water`、`light`、`leaf_spot`、`pest` |
| `allow_as_primary` | 是否可作为主方向 | `1` |
| `allow_as_secondary` | 是否可作为伴随观察方向 | `1` |
| `display_eligible_default` | 默认是否允许前端展示 | `1` |
| `is_intermediate_node` | 是否只是中间节点 | `1` 用于黄叶分流 |
| `action_profile_key` | 行动建议模板键 | `action_overwatering_basic` |
| `action_conflict_group` | 行动冲突组 | `watering_stop`、`watering_add` |
| `risk_level` | 风险等级 | `low`、`medium`、`high` |
| `route_review_status` | route 审核状态 | `draft`、`audited` |

如果暂不改表，可以先把这些信息放到新表 `diagnosis_outcomes`。

## 四、Phase 2：新增 route group / route / gate 核心表

### 1. `outcome_route_groups`

中文名：分流路径组表。

用途：定义同一入口场景下需要并行比较的一组候选 outcome。黄叶、萎蔫、焦边、叶斑、孔洞都应优先建 route group，而不是一开始指向某个单独 outcome。

建议字段：

| 字段 | 中文含义 | 类型建议 | 示例 |
|---|---|---|---|
| `route_group_key` | 分流路径组键 | varchar | `yellowing_care_split_group` |
| `route_group_name_cn` | 中文名 | varchar | `黄叶养护分流组` |
| `entry_scene_cn` | 入口场景说明 | text | `叶片发黄、老叶黄、新叶淡黄` |
| `entry_symptom_keys_json` | 入口症状 | json/text | `["leaf_yellowing"]` |
| `candidate_outcome_keys_json` | 候选 outcome | json/text | `["overwatering_root_pressure", "underwatering_dehydration_pressure"]` |
| `default_split_question_key` | 默认分流问题 | varchar | `q_soil_moisture_recent` |
| `max_visible_outcomes` | 最多前端可见方向 | number | `3` |
| `visible_question_purpose_cn` | 问题目的说明 | text | `用于区分积水、缺水、自然老叶和光照不足。` |
| `enabled` | 是否启用 | number | `1` |
| `review_status` | 审核状态 | varchar | `audited` |

| `entry_symptom_keys_json` | 入口症状 | json/text | `["leaf_yellowing"]` |
| `entry_direction_keys_json` | 入口诊断方向 | json/text | `["yellowing_direction"]` |
| `entry_symptom_class_keys_json` | 入口症状模式 | json/text | `["yellowing_mode"]` |

### 3. `outcome_route_gates`

用途：定义一条 route 或一个候选 outcome 的展示、增强、削弱、闭合、阻断、冲突和不确定条件。

建议字段：

| 字段 | 中文含义 | 类型建议 | 示例 |
|---|---|---|---|
| `gate_key` | gate 键 | varchar | `wet_soil_confirmation_gate` |
| `route_key` | 所属路径 | varchar | `yellowing_wet_soil_route` |
| `gate_role` | gate 角色 | varchar | `split`、`confirmation`、`display`、`blocker`、`closure`、`action_safety` |
| `required_evidence_json` | 必需证据 | json/text | `{ "symptomKeys": ["leaf_yellowing"] }` |
| `required_answer_effects_json` | 必需回答效果 | json/text | `{ "support": ["soil_long_wet"] }` |

| `conflict_outcome_keys_json` | 冲突 outcome | json/text | `["underwatering_dehydration_pressure"]` |
| `closure_level` | 闭合强度 | varchar | `confirmed`、`probable`、`suggestive` |
| `on_pass` | 通过后动作 | varchar | `strengthen_outcome`、`display_outcome`、`close_primary`、`close_secondary`、`ask_next` |

## 八、数据审核规则

每次导入 route 数据前必须检查：

1. 每个 final outcome 至少有一个启用 route。
2. 每条 route 至少有一个 closure gate 或 uncertain gate。
3. 每条需要问诊的 route 至少有一个 question。
4. 每个 question 的 option 都有明确 effect 或允许 neutral。
5. 每个 final outcome 必须有 action profile。
6. 同一个 action profile 不得包含互相冲突动作。
7. 中间节点不得被标记为 final output。
8. 未审核 route 不得在生产启用。

## 十、最小可行数据集

第一阶段不要把所有表都填满。最小可行：

```text
10 个 outcome
20～30 条 route
20～40 个 gate
20～30 个 route question
每个可输出 outcome 至少 1 个 action profile；多 outcome 同屏展示时，必须先通过行动建议冲突检查
```

这已经足够替代大部分 ranking 终局逻辑。
