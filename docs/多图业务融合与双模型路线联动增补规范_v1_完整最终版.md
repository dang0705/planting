# 多图业务融合与双模型路线联动增补规范_v1_完整最终版

> 状态：完整最终版  
> 定位：A 类正式基线联动增补候选稿  
> 主题：在既有“多图业务融合”裁决基础上，进一步把 **混元模型复用** 与 **HF AutoTrain 自训练模型过渡** 纳入同一套运行时、提示词、颗粒度、证据接纳与诊断更新框架。  
> 适用范围：视觉入口层、运行时模型、核心数据结构、诊断硬约束、决策流、AI 视觉实现方案、提示词与模型适配层。

---

## 0. 文档目标

本文件解决的不是“是否做多图”，而是以下四个已经进入工程期的实际问题：

1. **MVP 阶段以混元模型为主**，但其输出不能与既有体系脱节，必须服从统一的多图业务融合逻辑。  
2. **过渡阶段将逐步引入 HF AutoTrain 自训练模型**，但当前阶段其训练范式明确为 **image classification 的单样本单图**，因此必须通过业务层与运行时层完成多图融合。  
3. 无论上游视觉能力来自 **混元模型** 还是 **HF 自训练模型**，下游都必须进入**同一套标准化对象、同一套接纳判定、同一套 evidence / diagnosis 主链**。  
4. 为避免未来“模型切换即重写业务”，必须现在就冻结一套**模型无关、供应商无关、训练路线无关**的视觉输出 contract 与多图融合 contract。  

本文件的核心结论是：

> **多图业务融合是平台级能力，不是某个模型专属能力。**  
> 当前阶段，混元模型与 HF AutoTrain 模型都必须复用同一套多图融合逻辑；区别只允许存在于“上游识别来源”，不允许传导为“下游业务语义漂移”。

## 3. 上位原则（新增双模型约束）

### 3.1 模型无关原则

无论视觉结果来自：

- 混元多模态模型
- HF AutoTrain 自训练单图模型
- 后续其他模型供应方

都必须先被投影到**统一的视觉标准化对象**中，方可进入业务主链。

这意味着：

- **业务不能依赖某一供应商特有字段**
- **evidence 不能依赖某一模型特有置信表示**
- **prompt 不能直接变成业务语义**
- **模型升级不能改变 diagnosis 层字段含义**

### 3.2 视觉上游 / 业务下游隔离原则

视觉模型只负责回答以下类型的问题：

- 这张图更像哪个器官 / 部位
- 这张图可见哪些视觉症状
- 这些症状各自强弱如何
- 是否建议补图
- 是否存在明显图像质量问题
- 是否存在器官冲突或图像不适用

视觉模型不得越权回答：

- 最终 diagnosis 是什么
- 用户现在该执行哪条动作
- 该问题在 taxonomy 中的最终归属
- 哪个 problem key 已经正式成立

### 3.3 颗粒度统一原则

混元模型与 HF 自训练模型在上游识别阶段，可以采用不同的内部实现方式，但对外必须对齐到同一套“视觉标准化颗粒度”。当前阶段正式冻结的对齐方向为：

- **识别对象：视觉症状，而不是最终病因**
- **识别单位：可见征象，而不是解释后的 diagnosis**
- **结构化输出：标准化症状键 + 器官 + 强弱 + 质量 + 来源**

换言之，当前阶段不能让混元输出“根腐病”，而 HF 输出“黄化、萎蔫、根黑”；也不能让混元按粗粒度“叶片异常”，而 HF 按细粒度“叶脉间失绿”。如果两边颗粒度不一致，下游就无法稳定聚合。

## 4. 当前阶段双模型技术格局

### 4.1 MVP 阶段：混元主识别

MVP 阶段以混元模型为主，原因在于：

- 其多模态理解能力适合快速覆盖复杂输入场景
- 对零样本 / 小样本的开放场景鲁棒性更适合冷启动
- 能更快承接多图、补图、追问式输入

但这里必须强调：

> **MVP 阶段“以混元为主”，不等于“业务语义依赖混元输出格式”。**

混元只能是当前主要视觉来源，不是当前业务语义的定义者。

### 4.2 过渡阶段：HF AutoTrain 逐步接管稳定症状识别

过渡阶段，HF AutoTrain 自训练模型的角色不是直接替换整个视觉链路，而是逐步接管那些：

- 颗粒度稳定
- 边界较清晰
- 图像可见性强
- 标注可持续积累
- 单图即可判断主征象

的视觉症状类别。

例如：

- 黄化
- 褐斑
- 白粉覆盖
- 孔洞 / 虫咬
- 叶尖焦枯
- 叶片卷曲
- 可见腐烂征象
- 水渍状斑块

不宜一开始就用 HF 单图模型接管的内容包括：

- 需要跨器官综合判断的问题
- 需要强上下文解释的问题
- 图像可见症状与真实病因距离太远的问题
- 严重依赖多轮问答纠偏的问题

### 4.3 最终阶段：双模型并存但业务层不分叉

即使未来 HF 模型逐渐成熟，也不应把系统写成：

- 混元一套业务流
- HF 一套业务流

正确做法是：

- 上游有多个视觉来源
- 中间统一标准化与聚合
- 下游只有一套 evidence / diagnosis 主链

因此，本文件正式禁止“按模型分业务流”。

## 5. 双模型统一输出 contract（新增冻结）

为保证混元与 HF 的输出可替换、可聚合、可评审、可回流，当前阶段必须冻结统一的视觉输出 contract。所有模型适配器都必须以此为目标对象。

### 5.1 单图标准化结果对象（统一）

建议统一为 `visual_normalized_image_result`，最小必备字段如下：

| 字段 | 含义 | 说明 |
|---|---|---|
| `image_id` | 单图唯一标识 | 来源于当前批次 |
| `visual_call_batch_id` | 所属视觉调用批次 | 支持多图聚合与补图追踪 |
| `source_model_provider` | 来源模型供应方 | `hunyuan` / `hf_autotrain` / 其他 |
| `source_model_name` | 来源模型名 | 保留审计能力 |
| `input_organ_hint` | 输入器官槽位提示 | 由前端或用户提供，属于强提示 |
| `normalized_organ` | 标准化器官结果 | 视觉识别后标准化 |
| `image_quality_grade` | 图像质量等级 | 进入质量约束与补图逻辑 |
| `analyzability` | 可分析性 | 决定能否进入后续接纳 |
| `symptom_candidates` | 视觉症状候选列表 | 当前阶段核心输出 |
| `route_hints` | 流程路由提示 | 仅供流程层，不是事实层 |
| `suggested_followup_capture` | 建议补图列表 | 例如“补根部近景图” |
| `normalization_notes` | 标准化备注 | 记录冲突、歧义、降级原因 |

其中 `symptom_candidates` 的每个元素，最小建议字段如下：

| 子字段 | 含义 |
|---|---|
| `symptom_key` | 标准化视觉症状键 |
| `display_name_cn` | 中文展示名 |
| `strength_level` | 强 / 中 / 弱 |
| `confidence_band` | 高 / 中 / 低（带状，不直接暴露模型 raw score） |
| `visibility_scope` | 局部 / 器官级 / 全株级 |
| `supporting_region_note` | 关键局部描述 |
| `admission_readiness` | 初始建议：ready / cautious / retain_only |

### 5.2 聚合结果对象（统一）

不论混元是否原生多图，不论 HF 是否单图识别，多图提交后都必须生成同一类对象：`visual_call_aggregate_result`。

建议至少包含以下字段：

| 字段 | 含义 |
|---|---|
| `visual_call_batch_id` | 当前批次 |
| `organ_coverage_summary` | 器官覆盖情况 |
| `effective_image_count` | 有效图数量 |
| `duplicate_view_groups` | 重复视角分组 |
| `aggregated_symptom_candidates` | 聚合后的视觉症状候选 |
| `organ_support_summary` | 每个候选症状获得了哪些器官支持 |
| `aggregate_quality_grade` | 聚合质量 |
| `aggregate_analyzability` | 聚合可分析性 |
| `suggested_followup_capture` | 建议补图 |
| `admission_ready_flag` | 是否允许进入接纳判定 |
| `aggregate_route_hints` | 聚合路由提示 |

注意：

> **聚合结果不是 evidence。**  
> 聚合结果只是 evidence 的上游候选层。

## 6. 混元模型的正式使用边界

### 6.1 允许混元做的事

混元模型当前阶段可以负责：

1. 单图视觉症状识别
2. 多图输入下的跨图观测辅助
3. 器官判定与器官冲突提示
4. 质量问题识别
5. 建议补拍哪个器官 / 哪个角度
6. 生成结构化的候选视觉症状列表
7. 生成受限的 route hint

### 6.2 不允许混元做的事

混元当前阶段不得直接在业务层面做以下事情：

1. 直接返回 diagnosis 结论并写入主链
2. 直接返回 outcome 并触发动作建议
3. 把 prompt 内部推理过程映射为正式 evidence
4. 以多图黑盒综合结果绕过单图与聚合标准化
5. 使用未经标准化的自由文本症状替代标准症状键

### 6.3 为什么即使混元支持多图，也不能直接黑盒一步到位

因为当前体系的主轴不是“拿到一个看似聪明的答案”，而是“**建立可解释、可追溯、可补图、可纠错、可迁移的证据链**”。

如果允许混元直接黑盒输出最终 diagnosis，就会带来以下问题：

- 证据层失真
- 追问层失真
- 补图逻辑失真
- 自训练模型无法平滑接管
- 不同模型输出无法同构比较
- review、审计和回流修正难以成立

所以，本文件正式裁定：

> **混元的多图能力只能作为上游增强，不能作为业务层替代。**

## 7. HF AutoTrain 模型的正式使用边界

### 7.1 当前阶段正式前提

当前阶段 HF 自训练模型的正式前提为：

- 训练平台：Hugging Face AutoTrain
- 任务类型：image classification
- 样本粒度：单样本单图
- 多图能力：业务层多图批次与聚合，不来自训练层

### 7.2 HF 模型当前最适合承担的职责

HF 当前阶段最适合承担：

- 高频稳定症状的单图识别
- 器官内局部可见征象识别
- 逐步替换混元在稳定类别上的单图判断
- 作为结构化、低延迟、低成本的稳定基线

### 7.3 HF 模型当前不应承担的职责

HF 当前阶段不应承担：

- 原生单样本多图病例联合识别
- 强上下文多轮问答
- 多器官联合病因推断
- 复杂自由文本解释生成

因此，HF 模型的地位应被写成：

> **单图视觉症状识别器，而不是当前阶段的终局诊断器。**

## 8. 双模型统一颗粒度规范（核心新增）

这是本文件最关键的收口之一。若不冻结这一层，混元与 HF 永远无法共用一套多图融合逻辑。

### 8.1 当前阶段统一识别颗粒度

当前阶段统一识别颗粒度必须设为：

> **“视觉症状层”而不是“最终问题层”**

推荐的层级关系：

- **器官层**：叶 / 茎 / 根 / 根颈 / 全株 / 花 / 果 / 其他
- **视觉症状层**：黄化、褐斑、白粉覆盖、卷曲、焦边、穿孔、腐烂征象、萎蔫征象、水渍状斑块、畸形等
- **evidence 层**：接纳后的观测证据
- **diagnosis 层**：基于多证据综合计算后的问题候选
- **outcome / action 层**：对用户展示与动作建议

### 8.2 颗粒度统一规则

1. **优先统一到中等稳定颗粒度**，不要过粗，也不要过细。  
2. **先统一中文主名和标准 symptom_key**，再允许各模型内部维持自己的 raw label。  
3. 对于混元自由文本输出，必须经过**标准化映射层**，映射失败只可 retain，不得直接 admitted。  
4. 对于 HF 标签体系，训练集标签必须尽量与标准 symptom_key 一一对应，避免出现“训练标签和标准键不是同一物”的情况。  
5. 任何模型若输出超出当前标准颗粒度的更细症状，只能作为 `normalization_notes` 或二级属性保留，不得先于主颗粒度进入 evidence。  

### 8.3 当前阶段不建议直接让模型识别的对象

以下类型当前阶段不宜作为模型直接目标标签：

- 解释型病因键
- 需要跨器官与强上下文才能判断的问题
- 明显混合了视觉、环境、养护历史的综合概念
- 需要多轮确认才可成立的诊断项

## 9. 双模型统一提示词 / 适配层规范（新增）

### 9.1 提示词不是业务协议

混元阶段不可避免会使用 prompt，但必须正式写明：

> **Prompt 是模型适配实现，不是业务协议。**

业务协议永远是：
- 标准化字段
- 标准化症状键
- 标准化 admission / retain 边界
- 标准化 route hint 作用域

### 9.2 混元提示词的正式目标

混元提示词必须围绕以下目标构造，而不是追求“回答得像医生”：

1. 识别器官
2. 识别可见症状
3. 识别图像质量问题
4. 判断是否适合进入下一步
5. 生成有限的补图建议
6. 输出受限的结构化候选

### 9.3 混元提示词禁止目标

混元提示词中必须避免引导模型：

- 直接断定最终病因
- 输出最终治疗方案
- 把推测写成事实
- 产出未受控的 taxonomy 文本
- 产生过宽泛的大段解释替代结构化字段

### 9.4 提示词输出的建议层级

建议混元提示词输出三层信息：

#### A. 图像级判断
- 器官
- 质量
- 是否可分析

#### B. 症状级候选
- 标准症状键候选
- 强弱
- 可见位置说明
- 置信带

#### C. 流程级建议
- 是否建议补图
- 建议补什么器官
- 是否建议继续 follow-up
- 是否建议保守降级

其中 C 层只能进入流程层，不能直接成为 evidence。

### 9.5 HF 模型不使用 prompt，但必须复用同一标准化层

HF 模型虽然不依赖 prompt 产生结果，但输出后也必须经过与混元一致的标准化层：

- raw label → standard symptom_key
- raw confidence → confidence_band
- model output → admission_readiness suggestion
- image-level result → visual_normalized_image_result

因此，**混元有 prompt，HF 没 prompt，但二者共享同一标准化出口。**

## 10. 双模型共享的多图融合主线（正式冻结）

当前阶段，混元与 HF 都必须服从以下统一主线：

### 步骤 1：单次提交形成 `visual_call_batch`

无论输入来自：

- 混元多图请求
- HF 多张单图并行识别

业务层都统一创建一个 `visual_call_batch`。

### 步骤 2：保留逐图结果

即使混元支持多图理解，也不得只保留总结果，必须保留逐图结果。因为：

- 多图聚合需要逐图依据
- 补图与覆盖需要逐图依据
- review 与回流需要逐图依据
- 未来 HF 接管时需要逐图对齐基准

### 步骤 3：生成统一 `visual_normalized_image_result`

每张图都必须被标准化到统一对象。

### 步骤 4：生成统一 `visual_call_aggregate_result`

多图聚合统一由业务层完成，而不是由模型内部黑盒定义。

### 步骤 5：接纳判定

只允许 admitted 的聚合症状进入 `observed_evidence_set`。其他仅保留为 candidate 或流程提示。

### 步骤 6：diagnosis 更新

diagnosis 只能消费 evidence，不直接消费模型 raw output。

## 11. 双模型共享的多图融合规则（补强版）

### 11.1 重复视角抑制规则

无论症状来自混元还是 HF，同一批次内同一器官同一局部的重复视角都不得线性累计支持度。重复出现只能提高稳定性，不能简单按图片数叠加。

### 11.2 跨器官主从规则

无论上游模型是谁，跨器官支持的主从关系保持不变：

- 根腐方向：根 / 根颈 > 茎基部 > 叶片
- 虫害方向：叶背 / 嫩梢 / 茎局部 > 全株泛黄
- 日灼方向：叶片局部 + 全株分布优先
- 花果问题：花果与连接部位优先

模型不同，不改变器官优先级。

### 11.3 质量不足优先补图规则

无论上游结果多“聪明”，若质量不足、器官缺失或冲突未解，系统优先补图或保守降级，不得强推 diagnosis。

### 11.4 新旧批次显式覆盖规则

新批次永远不能隐式吞掉旧批次。无论混元还是 HF，只要形成新批次，就必须记录：

- 覆盖对象
- 覆盖原因
- 覆盖范围
- 覆盖时间
- 覆盖来源

---

## 12. 运行时的双模型适配层设计

为避免业务层和模型层耦合，建议显式增加**视觉模型适配层**。

### 12.1 建议增加的适配器

- `hunyuan_visual_adapter`
- `hf_autotrain_visual_adapter`

两者都输出同一类对象：
- `visual_raw_result`
- `visual_normalized_image_result`

### 12.2 适配层职责

#### 混元适配器职责
- 组装受限 prompt
- 解析混元返回内容
- 映射到标准器官与标准 symptom_key
- 生成带状 confidence
- 写入 normalization notes

#### HF 适配器职责
- 接收单图分类结果
- 标签映射到标准 symptom_key
- 把 raw score 映射到 confidence_band
- 补足 route_hint 缺省值
- 生成与混元同构的标准化结果

### 12.3 适配层不得承担的职责

适配层不得：

- 直接 admitted evidence
- 直接写 diagnosis
- 直接输出面向用户的最终解释
- 越权决定 supersede 关系

这些都属于业务运行时层。

## 13. 迁移路线：从混元主导到 HF 逐步接管

### 13.1 迁移总原则

迁移不是“从 A 系统切到 B 系统”，而是“上游模型替换，但中下游 contract 不变”。

### 13.2 当前阶段建议路线

#### 阶段 A：混元为主，HF 为空或灰度
- 混元承担大部分视觉识别
- 运行时开始沉淀标准化对象
- 开始积累按标准 symptom_key 标注的训练样本

#### 阶段 B：HF 接管稳定单图症状
- 对稳定、高频、单图可判类目引入 HF 模型
- 混元继续承担复杂、多图、弱结构、开放场景输入
- 双模型并存，但通过统一 contract 向下游输出

#### 阶段 C：HF 覆盖率逐步上升
- 更多稳定视觉症状由 HF 接管
- 混元退居复杂边界样本、解释辅助、补图建议
- 业务层不发生分叉

### 13.3 迁移期间的灰度比较原则

同一张图、同一批次、同一标准颗粒度下，可允许混元与 HF 同时产出结果用于离线比对。但在主链中，必须明确：

- 谁是主识别源
- 谁是 shadow / compare 源
- shadow 结果不得直接污染主 diagnosis

## 14. 需要联动增补的既有文档（本文件给出明确指令）

### 14.1 视觉入口数据结构与留存规范

必须新增或强化：

- `source_model_provider`
- `source_model_name`
- 双模型同构 `visual_normalized_image_result`
- 混元多图与 HF 多单图共享同一 `visual_call_batch`
- 逐图结果必须保留

### 14.2 运行时模型

必须新增或强化：

- 模型适配层概念
- `origin_visual_call_batch_id`
- `source_model_provider`
- `superseded_by_batch_id`
- shadow / compare 结果不得直接写主 evidence

### 14.3 核心数据结构

必须新增或强化：

- 标准视觉症状键与显示名的冻结映射
- `duplicate_view_group` 与 `independence_group_id` 的层级区分
- `admission_readiness`
- `confidence_band` 的统一语义

### 14.4 diagnosis hard constraints

必须新增或强化：

- 供应商无关原则
- 模型 raw output 不得直写 diagnosis
- 质量不足优先补图
- 多图重复不线性加分
- 模型切换不得改变 diagnosis 语义

### 14.5 决策流

必须显式补入：

输入  
→ 模型适配  
→ 单图标准化  
→ 多图聚合  
→ 接纳判定  
→ evidence 入集  
→ diagnosis 更新  
→ 解释 / 追问 / 补图

## 15. 当前阶段明确不允许的误实现

1. 因为混元支持多图，就绕过单图留存与聚合对象  
2. 因为 HF 是自训练模型，就单独走一套 evidence 语义  
3. 把混元 prompt 输出的自由文本直接当正式证据  
4. 让混元输出 diagnosis，而 HF 只输出 symptom  
5. 让 HF 标签体系与标准 symptom_key 长期错位共存  
6. 用模型供应方区分业务流  
7. 在迁移期间直接用 shadow 模型结果污染主链  
8. 把器官槽位提示当作视觉事实  
9. 因为多图提交就线性倍增支持度  
10. 因为模型表现强，就跳过补图和保守降级逻辑

---

## 16. 最终绑定决策（双模型版）

1. **绑定决策 6**：当前阶段多图业务融合属于平台级能力，混元模型与 HF AutoTrain 模型必须共享同一套多图融合逻辑。  
2. **绑定决策 7**：MVP 阶段以混元模型为主，但混元只定义上游识别来源，不定义业务语义。  
3. **绑定决策 8**：当前阶段 HF 自训练模型正式限定为 AutoTrain image classification 的单图症状识别模型。  
4. **绑定决策 9**：混元与 HF 的识别结果必须统一对齐到同一套视觉症状颗粒度与标准化 contract。  
5. **绑定决策 10**：所有上游模型结果都必须先经过适配层、标准化层、聚合层、接纳层，再进入 evidence / diagnosis 主链。  
6. **绑定决策 11**：未来模型切换只允许发生在上游识别来源，不允许引发中下游业务语义分叉。

## 18. 最终结论

在加入“**MVP 以混元为主，后续逐步切换到 HF 自训练模型**”这一现实前提后，当前体系不能只写“HF 单图 + 多图业务融合”，而必须升级为：

> **混元主导冷启动识别，HF 逐步接管稳定单图症状识别；两者共享统一的视觉标准化 contract、统一的多图聚合逻辑、统一的 evidence 接纳规则、统一的 diagnosis 主链。**

这样做的价值在于：

- 当前 MVP 能快
- 后续过渡能稳
- 模型切换不推翻业务
- 宪法、定义、数据层与运行时层保持统一
- 多图能力不再绑死在某个供应商身上

本文件因此正式建议：下一步不再单独写“混元方案”和“HF 方案”，而是统一落成 **《视觉模型适配层与双模型多图融合实施规范》**，并同步增补视觉入口、运行时模型、核心数据结构、diagnosis hard constraints 与决策流。
